## 단일 서버

모든 컴포넌트가 단 한 대의 서버에서 실행되는 간단한 시스템부터 설계해보자.

<!-- ![](images/p01-002.png) -->

이 이미지의 구성을 이해하기 위해 사용자의 요청이 처리되는 과정과 요청을 만드는 단말에 대해 이해해보자.

### 사용자의 요청 처리 흐름

<!-- ![](images/p01-003.png) -->

`www.google.com`에 접속한다고 해보자.

1. 사용자는 도메인 네임인 `www.google.com`을 이용해 웹사이트에 접속한다.
   - 이 과정에서 DNS(Domain Name Service)에 질의하여 IP 주소로 변환하는 과정이 필요하다.
     <!-- TODO: DNS의 정식 명칭은 Domain Name System. 책(Vol.1 1장)에도 "도메인 네임 시스템"으로 표기. -->
   - 참고로 DNS는 보통 제 3사업자가 제공 (ex. 구글, cloudflare, amazon 등)
2. DNS 조회 결과 IP 주소가 반환된다. 이게 웹 서버의 주소.
3. 해당 IP 주소로 HTTP 요청이 전달됨.
4. 웹 서버는 HTML 페이지나 JSON 형태의 응답 반환.

### 요청을 만드는 단말: 웹 앱 vs 모바일 앱

**웹 앱**
- 인터넷 브라우저를 통해 실행되는 앱. 별도의 설치가 필요 없다. URL을 통해 접근.
- ex. 구글 docs, 구글 sheets, gpt, notion 등

**모바일 앱**
- 일반적으로 앱스토어에서 설치가 필요.
- 특정 운영체제에 맞는 개발이 필요. 아니라면 크로스 플랫폼 프레임워크를 이용해야 함.
- ex. notion, gpt, x, instagram 등

#### 웹 앱과 모바일 앱의 서버 요청은 다른가?

**공통점**
- HTTP/HTTPS 프로토콜을 기반으로 데이터를 주고받음
- RESTful API, GraphQL 같은 표준화된 인터페이스를 통해 통신
- 서버와의 데이터 교환은 주로 JSON

**차이점 — 요청 방식 및 프로토콜**

| 구분 | 웹 앱 | 모바일 앱 |
|---|---|---|
| 요청 수단 | 브라우저의 Fetch API / XMLHttpRequest | 네이티브 라이브러리 (OkHttp, Retrofit, URLSession) |
| 네트워크 감지 | 제한적 | 셀룰러/Wi-Fi 상태 직접 감지·최적화 가능 |
| 예시 | `fetch('https://api.example.com/data', { method: 'GET' })` | `OkHttpClient().newCall(request).execute()` |

**차이점 — 데이터 처리 및 캐싱**

| 구분 | 웹 앱 | 모바일 앱 |
|---|---|---|
| 저장소 | LocalStorage, IndexedDB, Service Worker 캐시 | 파일 시스템, 로컬 DB (Room, Core Data) |
| 크기 제한 | 보통 5–10MB, 브라우저 정책에 따라 삭제 가능 | 훨씬 유연 |

## 데이터베이스

단일 서버 서비스에서 사용자가 늘었다면:

> 여러 서버를 두자. 하나는 웹/모바일 트래픽 처리 용도. 하나는 데이터 저장 용도.

그렇다면 어떤 데이터베이스를 고를까?

### 관계형 vs 비관계형

**관계형 데이터베이스 (RDBMS)**
- 테이블과 열, 칼럼으로 자료를 표현.
- 왜 이름이 relational 일까? → https://www.youtube.com/watch?v=gjcbqZjlXjM

**비관계형 데이터베이스 (NoSQL)**
- 관계형 데이터베이스의 문제점들을 보완하기 위해 나온 데이터베이스.
- ACID 원칙 일부를 포기함으로써 성능을 끌어올렸다. → https://www.youtube.com/watch?v=sqVByJ5tbNA

## 수직적 규모 확장 vs 수평적 규모 확장

> 사용자가 더 많이 늘어나서 웹 트래픽을 기존의 컴퓨터가 부담하기 힘들어졌다… 어떻게 할까?

2가지 방법이 있다.

- **수직적 규모 확장**: 기존의 컴퓨터를 업그레이드한다.
- **수평적 규모 확장**: 여러 컴퓨터를 사용해서 사용자들을 분산시킨다.

트래픽 양이 그리 많지 않을 때는 수직적 확장이 좋은 선택. 그러나 단점이 있다.

1. 수직적 규모 확장엔 한계가 있음. 한 대의 서버에 CPU나 메모리를 무한대로 증설할 수 없다.
2. 장애에 대한 자동 복구 방안이나 다중화 방안을 제시하지 않는다.

이러한 문제를 수평적 규모 확장을 통해 해결하자.

### 로드밸런서

> 수평적 규모로 확장한다면 여러 컴퓨터에 유저들을 골고루 보내야 할 텐데 어떻게 할까? → **로드밸런서**가 해결한다.

<!-- ![](images/p03-004.png) -->

1. 사용자는 로드밸런서의 공개 IP 주소로 접속
2. 로드밸런서가 사설 IP 주소를 이용하여 부하를 나눈다.

이와 같은 구조는 다음 문제들이 생겼을 때 해결할 수 있다.

- **서버1 다운** → 서버2로 모든 트래픽 전송
- **트래픽 증가** → 서버1, 서버2로 감당 못하면 서버 추가만 하면 됨

이제 웹 계층은 괜찮아 보이는데, **데이터 계층이 문제다.**

### 데이터베이스 다중화

지금까지는 데이터베이스가 1개였는데 그러면 문제가 뭘까?
- 1개가 다운되면 서비스 사용 불가.
- 심각한 손상으로 복구 불가능하다면? 지금까지의 데이터는 없다…

이러한 불상사를 해결하기 위해 데이터베이스도 다중화를 한다. 보통은 **주 데이터베이스(master DB)** 와 **부 데이터베이스(slave DB)** 를 만든다.

| 역할 | 책임 |
|---|---|
| 주 데이터베이스 | 데이터 변경은 주에서만. 부 데이터베이스로 사본 넘김. |
| 부 데이터베이스 | 주로부터 사본을 전달받음. 읽기만 지원. |

보통은 읽기 연산이 훨씬 많으므로 다음과 같은 구조가 된다.

<!-- ![](images/p04-005.png) -->

> 주 데이터베이스는 1개이고 부 데이터베이스를 여러 개 가진다.

이런 식으로 만들면:
- 읽기 연산이 분산되므로 성능이 좋아진다.
- 서버 가운데 일부가 파괴되어도 데이터가 보존된다.
- 데이터베이스 하나가 망가져도 계속 서비스가 가능하다.

#### 실제 장애 시나리오

1. **부 서버가 1대인데 다운됨** → 한시적으로 주가 부의 역할을 맡음
2. **부 서버가 여러 대인데 1개 다운** → 다른 부 서버가 장애 서버 연산을 커버
3. **주 서버 다운** → 부 서버가 1대면 부가 주의 역할도 맡음. 그리고 새로운 부 서버를 추가하면 됨.

> 3번의 경우는 실제 프로덕션 환경에서 문제가 있다. 주가 망가지면서 주와 부의 데이터 불일치가 나타날 수 있기 때문이다. 주에서는 데이터가 바뀌었는데 부에서는 안 바뀐 것. 해결방법은 복구 스크립트를 돌려서 추가해야 된다고 한다.

## 캐시

> 값비싼 연산 결과나 자주 참조되는 데이터를 메모리에 저장하여 뒤이은 요청이 보다 빨리 처리될 수 있도록 하는 저장소.

앱의 성능은 데이터베이스를 얼마나 자주 호출하느냐에 크게 좌우되는데, 캐시가 그 문제를 해결해준다.

<!-- ![](images/p05-007.png) -->

> 위의 방식을 **읽기 주도형 캐시 전략(read-through)** 이라고 한다. 이것 외에도 다양한 캐시 전략이 있다.

### 캐시 사용 시 유의할 점

1. 데이터 갱신은 자주 일어나지 않지만 참조가 빈번하게 일어나면 고려해볼 만하다.
2. 영속적으로 보관해야 되는 데이터는 캐시에 두면 안 된다.
3. **만료 정책**: 캐시의 데이터는 언제 만료되는가. 이에 대해서 정책을 마련해두어야 한다.
4. **일관성**: 저장소의 원본 갱신 연산과 캐시를 갱신하는 연산이 단일 프로세스(트랜잭션)로 일어나지 않으면 일관성 유지가 어렵다.
5. **장애 대비**: 캐시 서버가 1개인데 장애 나면? 캐시 서버도 분산시켜야 된다.
6. **메모리 크기**: 과할당 해놓는 것이 좋아 보인다.
7. **데이터 방출 정책**: 캐시가 꽉 차면 어떻게 할 건가. 제일 많이 쓰이는 방법은 **LRU** (마지막으로 사용된 시점이 가장 오래된 데이터를 내보내는 것).

> 캐시 말고는 응답 시간을 줄일 방법이 없나? 물리적으로 서버가 내 근처에 있으면 되지 않을까?

### CDN 사용 시 고려사항

- CDN으로 들어가고 나가는 데이터 전송 양에 따라 요금을 내므로 자주 사용하지 않는 것은 빼자.
- 만료 기한도 적절히 선정해야 한다.
- CDN 자체가 죽었다면 해당 문제를 감지하여 원본 서버로부터 직접 콘텐츠를 가져오도록 구성하는 것이 필요.
- 아직 만료되지 않은 콘텐츠 제거 방법:
  - CDN 사업자가 제공하는 API 사용
  - 콘텐츠의 다른 버전을 서비스 하도록 **오브젝트 버저닝** 이용. URL 마지막에 버전 번호를 인자로 주면 됨.

## 무상태 웹 계층

> 웹을 수평적으로 확장하기 위해 상태 정보(사용자 세션 데이터와 같은)를 제거해야 한다. 바람직한 전략은 상태 정보를 데이터베이스에 저장하고 필요할 때 가지고 오는 것. 이러한 전략을 **무상태 웹 계층**이라 부른다.

무상태 말고 상태 정보 의존적인 아키텍처는 뭘까? 책과는 다른 예시를 들어보자. (김영한 님 강의에서 가져왔다.)

### Stateful 예시

```
고객 : 이 노트북 얼마인가요?
점원 : 100만원입니다.
고객 : 2개 구매할게요
점원 : 200만원 입니다. 신용카드, 현금 중 어떤 걸로 결제하시겠어요?
고객 : 신용카드로 구매하겠습니다.
점원 : 200만원 결제 완료되었습니다.
```

**갑자기 점원이 중간에 바뀐 경우** → 망함:

```
고객   : 이 노트북 얼마인가요?
점원A : 100만원입니다.
고객   : 2개 구매하겠습니다.
점원B : 무엇을 2개 구매하시나요?
고객   : 신용카드로 구매하겠습니다.
점원C : ???
```

### Stateless 예시

```
고객   : 이 노트북 얼마인가요?
점원A : 100만원입니다.
고객   : 노트북 2개 구매하겠습니다.
점원B : 노트북 2개는 200만원입니다. 신용카드, 현금 중에 어떤 걸로 구매하시겠어요?
고객   : 노트북 2개를 신용카드로 구매하겠습니다.
점원C : 200만원 결제되었습니다.
```

### 정리

| | Stateful | Stateless |
|---|---|---|
| 점원 교체 | 불가능 | 가능 |
| 서버 증설 | 어려움 | 무한히 가능 |
| 응답 서버 변경 | 어려움 | 쉬움 |

<!-- ![](images/p07-011.png) -->

이제 서비스가 더 인기가 많아져 전 세계 사용자들이 이용한다고 하자. 이때는 **여러 데이터 센터**를 지원하는 것이 필수다.

## 데이터 센터

> 장애가 없는 상황에서 사용자는 가장 가까운 데이터 센터로 안내된다. 이를 **지리적 라우팅(geoDNS-routing)** 이라고 부른다.

지리적 라우팅에서 geoDNS는 사용자의 위치에 따라 도메인 이름을 어떤 IP 주소로 변환할지 결정할 수 있도록 해주는 DNS 서비스이다.

<!-- ![](images/p08-012.png) -->

### 다중 데이터센터의 기술적 난제

- **트래픽 우회**: 올바른 데이터 센터로 트래픽을 보내는 효과적인 방법을 찾아야 한다.
- **데이터 동기화**: 데이터 센터마다 별도의 데이터베이스를 사용한다면 트래픽이 우회되도 해당 데이터에는 데이터가 없을 수 있다. 보편적 전략은 데이터를 여러 데이터 센터에 걸쳐 다중화해 놓는 것.
- **테스트와 배포**: 여러 데이터 센터 위치에서 테스트를 해봐야 한다.

시스템을 더 큰 규모로 키우기 위해 시스템의 컴포넌트를 분리하자. 이때 **메시지 큐**가 쓰인다.

---

## 메시지 큐

> 메시지의 무손실을 보장하는 비동기 통신을 지원하는 컴포넌트. 메시지의 버퍼 역할을 하며 비동기적으로 전송한다.

기본 아키텍처는 간단하다.

<!-- ![](images/p09-014.png) -->

1. **생산자(producer)** 또는 **발행자(publisher)** 가 메시지 큐에 메시지 발행
2. 큐에는 **소비자/구독자(consumer/subscriber)** 라 불리는 서비스/서버가 연결되어 있는데 메시지를 받아 그에 맞는 동작을 수행

### 왜 메시지 큐를 쓰면 서비스 간 결합이 느슨해지나?

서비스 A와 서비스 B가 있다고 하자. A가 B에게 서버 요청을 한다.

- **강하게 결합되어 있다면**: A는 B가 응답을 보낼 때까지 아무것도 못 한다.
- **메시지 큐를 쓰면**: A가 요청을 보내고 기다리지 않아도 된다. 메시지 큐가 메시지를 보관하고 순차적으로 보내주기 때문이다.

> 그렇기에 생산자와 소비자는 각기 독립적으로 확장될 수 있다.

사업이 커졌으니 에러 로그와 메트릭도 보자.

---

## 로그, 메트릭 그리고 자동화

- **로그**: 에러 로그를 모니터링 하는 것은 당연히 중요. 서버마다 보는 것보단 한 번에 보는 게 나음.
- **메트릭**: 사업 현황에 관한 유용한 정보를 얻을 수 있고, 시스템 현 상황을 파악하기 쉽다.
  - **호스트 단위 메트릭**: CPU, 메모리, 디스크 I/O
  - **종합 메트릭**: 데이터베이스 계층의 성능, 캐시 계층의 성능
  - **핵심 비즈니스 메트릭**: 일별 능동 사용자, 수익, 재방문 등
- **자동화**: 시스템이 크고 복잡해지면 생산성을 높이기 위해 자동화 도구를 써야 함.

## 데이터베이스의 규모 확장

데이터베이스도 수직적 vs 수평적 확장이 있다.

### 수직적 확장

- 기존 서버에 고성능 자원을 추가하는 방법.
- ex. RAM을 24TB 짜리를 쓴다든가…

참고로 스택 오버플로우는 2013년 한 해 동안 천만 명의 사용자를 단 한 대의 마스터 데이터베이스로 처리했다고 한다. ㄷㄷ

### 수평적 확장

> 수직적 확장에는 한계가 있다. CPU나 RAM을 무한 증설할 수가 없다. 비용 문제도 있다.

- 수평적 확장은 **샤딩(sharding)** 이라고도 부른다.
- 더 많은 서버를 추가하는 것.
- 데이터베이스를 **샤드**라고 부르는 작은 단위로 분할한다.

<!-- ![](images/p10-017.png) -->

#### 샤딩 키

- 샤딩 전략에서 중요한 건 **샤딩 키**를 어떻게 정하느냐이다.
- 샤딩 키는 파티션 키라고도 부른다.
- 그림 1-22에서 샤딩 키는 `user_id`이다.

#### 샤딩의 문제점

데이터를 골고루 나눠줄 것 같은 샤딩 방식도 문제가 있다.

1. 샤드 키 계산 함수가 잘못된 경우 데이터가 하나의 데이터베이스로 쏠릴 수 있다.
2. 유명 인사들이 같은 샤드에 저장되면 과부하가 걸린다.
3. 여러 샤드에 걸친 데이터를 조인하기 힘들다.
