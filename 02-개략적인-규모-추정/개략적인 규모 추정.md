## 2의 제곱수

> 분산 시스템에서 다루는 데이터의 양에 대한 제대로 된 계산 결과를 얻으려면, 데이터 볼륨의 단위가 2의 제곱수로 어떻게 표현되는지 알아야 한다.

- 최소 단위는 **1바이트**이고, 8비트로 구성된다.
- ASCII 문자 하나가 차지하는 메모리의 크기는 1바이트.

## 모든 프로그래머가 알아야 하는 응답지연 값

> 응답지연 값은 시스템의 지연을 측정한 값으로, **지연 시간(Latency)** 이라고도 한다. 데이터가 요청한 컴퓨터에서 응답하는 컴퓨터에 도달하는 데 걸리는 시간을 말한다.

### 수치 분석 결과

- 메모리는 빠르지만 디스크는 아직도 느리다.
- 디스크 탐색은 가능한 한 피하라.
- 단순한 압축 알고리즘은 빠르다.
- 데이터를 인터넷으로 전송하기 전에 가능하면 압축하라.
- 데이터 센터는 보통 여러 지역에 분산되어 있고, 센터들 간에 데이터를 주고받는 데는 시간이 걸린다.

### 💡 메모리는 빠르고 디스크는 느린 이유

**하드웨어 구조 차이**
- **메모리(RAM)**: 전자 회로(반도체)로 구성 → 전기 신호만으로 데이터 접근
- **디스크(SSD/HDD)**: 데이터를 저장하고 검색하는 방식이 물리적인 기계 동작 포함

**데이터 접근 방식 차이**

- **직접 접근 (Direct Access)** — 메모리
  - CPU가 직접 데이터를 가져옴 (Load/Store 방식)
  - 메모리 셀에서 즉시 원하는 데이터를 읽고 쓸 수 있음
  - 메모리 주소를 바로 지정하여 접근하므로 랜덤 액세스(random access) 가능
- **블록 단위 접근** — 디스크
  - 디스크는 데이터를 블록(block) 단위로 저장
  - 원하는 데이터를 찾기 위해 여러 단계를 거쳐야 함

> **Jeff Dean의 키노트 메시지**: *"중요한 것은 실제로 만들지 않고도 시스템의 성능을 추정할 수 있는 능력이다."*

> **Norvig의 진짜 강조점**: 정확한 수치 암기가 아니라 **차수(order of magnitude)의 감각**. 나노초 / 마이크로초 / 밀리초 사이의 격차를 직관적으로 아는 것.

### 💡 단순한 압축 알고리즘?

> 데이터를 압축하여 작은 크기로 만드는 방법. 알고리즘이 간단하고 구현이 직관적인 방법을 사용.

**빠른 이유**
- **복잡한 계산이 적음**: 복잡한 수학적 연산을 피하고 간단한 탐색, 비교 또는 반복적인 패턴 인식으로 구현됨.
- **메모리 사용이 적음**: 압축 시에 메모리 사용이 최소화되어 빠른 처리 속도 유지.
- **적은 I/O 연산**: 디스크나 네트워크에서 데이터를 읽고 쓰는 I/O 연산을 최소화하는 방식.

## 레이턴시 수치를 어떻게 활용할까

> 이 수치들의 진짜 가치는 **시스템을 직접 만들기 전에 의사결정을 정량적으로 할 수 있게 해 준다**는 점.
>
> "느리다 / 빠르다"는 직관 대신, **차수와 배수로 객관적인 근거**를 댈 수 있어야 한다.

### 사례 1: 캐시 도입 의사결정

> **상황**: 주니어 개발자가 "DB 조회가 느려서 Redis 캐시 도입하고 싶습니다" 라고 함. 시니어가 "DB 응답이 얼마나 느린데요? 캐시로 얼마나 빨라지는데요?" 라고 물어보면 답할 수 있어야 한다.

수치를 인용해서 답해보자 (HDD 기반 레거시 DB 가정).

| 연산 | 레이턴시 | 단위 |
|---|---|---|
| HDD 디스크 탐색 | ~10 ms | **밀리초** |
| 데이터센터 내부 RTT (앱 → Redis) | ~500 μs | **마이크로초** |
| Redis 메모리 조회 | ~100 ns | **나노초** |

> **차수가 완전히 다르다**: 밀리초 → 마이크로초 → 나노초. 단위 자체가 1,000배씩 점프한다.

**결론**: HDD 조회 대비 Redis 조회가 압도적으로 빠르므로 **도입 가치가 충분**. 객관적인 수치 근거로 의사결정.

> ⚠️ **단, 실제 비교는 네트워크 RTT를 포함해서 봐야 한다.** 위 표는 단일 연산만 떼어 본 것이고, 앱 서버에서 DB·Redis까지 가는 경로에는 DC 내부 RTT 500μs가 항상 끼어든다.
>
> | 경로 | 구성 요소 | 대략 합계 |
> |---|---|---|
> | HDD DB 조회 | DC RTT 500μs + HDD 탐색 ~10ms | **~10ms** (디스크가 지배) |
> | SSD DB 조회 | DC RTT 500μs + SSD 랜덤 읽기 수십~수백 μs | **~600μs ~ 1ms** |
> | Redis 조회 | DC RTT 500μs + 메모리 ~100ns | **~500μs** |
>
> - **HDD ↔ Redis: 약 20배** — 응답시간 절감만으로도 캐시 도입 명분이 명확.
> - **SSD ↔ Redis: 1.3배 안팎** — 네트워크 RTT가 지배해서 응답시간 격차가 거의 사라진다.
>
> 즉 SSD 기반 환경에서는 **응답시간 이득이 아니라 DB QPS 부담 분산 / 핫키 보호 / 다운스트림 보호** 같은 관점으로 캐시 도입을 재검토해야 한다.

### 사례 2: 마이크로서비스 분리의 진짜 비용

> **질문**: 이 기능을 별도의 마이크로서비스로 분리하는 게 좋을까?

| 호출 방식 | 레이턴시 |
|---|---|
| 로컬 메모리 안에서 함수 호출 | 수 ns |
| 같은 데이터센터 내부 통신 | ~500 μs |

> **차이: 10만 배 이상.** 거기에 직렬화/역직렬화 비용까지 추가.

**결론**: 기능을 분리한다는 것은 호출 비용을 최소 10만 배 올리는 결정. 호출 빈도가 높다면 신중하게 접근해야 하고, **무조건 분리가 답은 아니다**.

> ⚠️ 물론 MSA 결정은 레이턴시만이 아니라 확장성·독립 배포·팀 경계 같은 아키텍처 관점도 함께 고려해야 한다. 여기서는 "수치로 비용을 정량화할 수 있다"는 점이 핵심.

### 사례 3: 100만 QPS 시스템 설계

> **면접 질문**: "URL 단축 서비스를 100만 QPS로 처리하려면 어떻게 설계하시겠어요?"

수치 감각이 있으면 다음 3가지가 즉시 떠올라야 한다.

1. **메모리 조회 ~100 ns**
   - 이론상 단일 서버에서 **초당 1천만 개**까지 처리 가능.
   - → 메모리에서 끝나는 작업이면 한 대로도 충분.
2. **디스크 DB 조회 ~수 ms**
   - 100만 QPS에서는 단일 서버 한계가 명확. 초당 수백 개 처리도 버겁다.
   - → **샤딩·캐시가 필수**.
3. **네트워크 호출 ~500 μs**
   - 서비스 간 호출 회수를 줄이지 않으면 병목.
   - → **호출 최소화** 설계 필요.

> 이렇게 "왜 캐시가 필수인가, 왜 샤딩이 필요한가, 왜 호출을 최소화해야 하는가"를 **숫자로 정량화해서 설명**할 수 있는 것이 코더와 아키텍트의 차이.

### 활용 시 주의점 4가지

1. **정확한 수치를 외우려 하지 마라.**
   - 이 수치들은 차수(order of magnitude) 수준의 **근사값**. 실제 환경에서는 정확히 일치하지 않을 수 있다.
   - 중요한 것은 **나노초 / 마이크로초 / 밀리초** 단위 차이의 감각.

2. **모든 수치를 다 외우지 말고 몇 개만 외워도 좋다.**
   - 다음 정도만 머릿속에 있어도 의사결정에 충분:
     - L1 캐시
     - 메인 메모리
     - SSD 랜덤/순차 읽기
     - 데이터센터 내부 RTT
     - 디스크(HDD) 탐색
     - 대륙 간 패킷 전송 비용

3. **절대값에 매달리지 마라.**
   - 연도마다 절댓값은 변할 수 있지만 **상대적 격차(배수)는 그대로 유지**된다.
   - 예: RAM이 SSD보다 약 200배 빠르다 — 이런 **비율 감각**이 중요.

4. **레이턴시만 전부가 아니다.**
   - 실무에서는 **스루풋(처리량)** 도 함께 고려해야 한다.
   - 배치·병렬·순차 처리로 레이턴시 비용을 상쇄할 수 있는 경우도 있다.
   - 예: 10개 호출을 하나로 묶으면 네트워크 왕복 비용이 1/10이 된다.

## 가용성에 관계된 수치들

> **고가용성**: 시스템이 오랜 시간 동안 지속적으로 중단 없이 운영될 수 있는 능력.

- 고가용성을 표현하는 값은 **퍼센트(%)** 로 표현
- 대부분의 서비스는 99% ~ 100% 사이의 SLA(Service Level Agreement)를 제공한다.

### SLA가 참조하는 가용성 지표 3가지

- **복구 시간 목표 (RTO, Recovery Time Objective)**: 계획된 중단에 소요되는 총 시간, 또는 계획되지 않은 중단에서 복구하는 데 소요되는 총 시간
- **평균 가동 중지 시간 (MDT, Mean Downtime)**: 시스템이 작동하지 않는 평균 시간
- **평균 무고장 시간 (MTBF, Mean Time Between Failures)**: 특정 시스템에서 다음 고장이 발생할 때까지 예상되는 시간

### 💡 서비스가 99% ~ 100%를 갖기 위해

- **시스템 이중화**: 서버나 구성 요소에 장애가 발생하더라도, 다른 구성 요소가 자동으로 역할을 대신
- **자동 복구 및 장애 조치**
- **데이터 백업 및 복구**
- **부하 분산**: 여러 서버에 트래픽을 균등하게 분배 (로드 밸런서)
- **모니터링 및 경고 시스템**

## 예제: 트위터 QPS와 저장소 요구량 추정

> 트위터와 같은 서비스의 QPS(Query Per Second), 즉 초당 쿼리 수를 추정하는 방법을 알아보자.
> (실제 트위터와는 관련 없음)

### 가정

- 월간 능동 사용자(MAU)는 **3억(300M)명**
- **50%** 의 사용자가 트위터를 매일 사용
- 평균적으로 각 사용자는 매일 **2건**의 트윗을 올린다
- 미디어를 포함하는 트윗은 **10%** 정도
- 데이터는 **5년**간 보관된다

### 추정 — QPS

- **DAU** (Daily Active User) = 3억 × 50% = **1.5억 (150M)**
- **QPS** = 하루 동안의 트윗 수 / 하루의 초 수
  - = 1.5억 × 2 트윗 / 24시간 / 3600초 ≈ **3,500**
- **Peak QPS** = 평균 QPS × 2 ≈ **7,000**

### 추정 — 미디어 저장소 요구량

**평균 트윗 크기**

| 항목 | 크기 |
|---|---|
| `tweet_id` | 64 바이트 |
| 텍스트 | 140 바이트 |
| 미디어 | 1 MB |

**계산**

- 미디어 저장소 요구량 / 일 = 1.5억 × 2 × 10% × 1MB = **30 TB / 일**
- 5년간 미디어 보관 = 30TB × 365 × 5 ≈ **55 PB**
