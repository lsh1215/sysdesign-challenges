# 시스템 설계 면접 공략법

> 실세계에서 사용하는 시스템의 구조는 극도로 복잡하다. 그런 시스템을 한 사람이 설계할 수 있으리라 아무도 기대하지 않는다.
> 그렇다면 시스템 설계 면접은 필요 없는 것 아닌가?
>
> → 모호한 문제를 풀며 그 사람의 설계 기술을 볼 수 있고, 내린 결정들에 대한 방어 과정도 볼 수 있다. 면접자의 생각의 과정이 얼마나 합리적인가를 볼 수 있는 것이다.

## 효과적인 면접을 위한 4단계 접근법

### 1단계: 문제 이해 및 설계 범위 확정

> 시스템 면접에서는 생각 없이 바로 답을 내서는 좋은 점수를 받기 어렵다. 답부터 들이밀지 마라. 속도를 늦춰라. 깊이 생각하고 질문하여 요구사항과 가정들을 명확히 하라.

**명심할 것**
1. 올바른 질문을 하라.
2. 적절한 가정을 하라.
3. 시스템 구축에 필요한 정보를 모아라.

**대표적인 질문 예시**
- 구체적으로 어떤 기능들을 만들어야 하나?
- 제품의 사용자 수는?
- 회사의 규모가 얼마나 빨리 커질 것인가?
- 회사가 주로 사용하는 기술 스택은?

#### 예제: 뉴스피드

- 모바일 앱과 웹 앱 가운데 어느 쪽을 지원해야 하는지? 둘 다 지원해야 하는지?
- 가장 중요한 기능은?
- 정렬 기준은 무엇인지 (가중치 같은 건 없는지)
- 최대 몇 명의 사용자와 친구를 맺을 수 있는지
- 사이트로 오는 트래픽 규모는 어느 정도인지 (DAU)
- 피드의 형태는 이미지나 비디오도 포함이 되는지? 단순 텍스트인지?

### 2단계: 개략적인 설계안 제시 및 동의 구하기

> 설계안을 제시하고 면접관의 동의를 얻기. (면접관과 협력하며 진행하면 좋음)

- **최초 청사진**을 제시하고 의견을 구하라. 면접관을 마치 팀원인 것처럼 대하라.
- 화이트보드나 종이에 핵심 컴포넌트를 포함하는 다이어그램을 그려라.
  - 클라이언트(모바일/웹), API, 웹 서버, 데이터 저장소, 캐시, CDN, 메시지 큐 같은 것들
- 최초 설계안이 시스템 규모 관련 제약사항을 만족하는지 **개략적으로 계산**해 보라. 소리를 내어 설명하라.
  - 단, 개략적 추정이 필요한지는 면접관에게 미리 물어보자.
- 가능하다면 시스템의 구체적 **사용 사례**도 몇 가지 살펴보자. 미처 고려하지 못한 에지 케이스(edge case)를 발견하는 데 도움이 된다.
- 이 단계에서 API 엔드포인트나 DB 스키마도 보여야 하는지는 면접관의 의견을 물어보자. 상황에 따라 지나치게 세부적일 수 있다.

#### 예제: 뉴스피드

대략적으로 아래 두 가지 처리 플로우로 나눠 생각해볼 수 있다.

1. **피드 발행**: 사용자가 포스트를 올리면 관련된 데이터가 캐시/데이터베이스에 기록되고, 해당 사용자의 친구(friend) 뉴스 피드에 뜨게 된다.
2. **피드 생성**: 어떤 사용자의 뉴스 피드는 해당 사용자 친구들의 포스트를 시간 역순으로 정렬하여 만든다.

**피드 발행 흐름**

<!-- ![](images/p02-002.png) -->

1. 사용자가 `/v1/me/feed?content=Hello&auth_token=auth_token` 형식으로 포스트를 올리는 요청을 보냄
2. DNS를 통해 요청이 웹 서버로 전달되고, 로드밸런서가 이를 적절한 원 서버로 분산함
3. 웹 서버는 포스트 데이터를 처리하여:
   - 포스팅 저장 서비스에 데이터를 저장
   - 포스팅 전송 서비스를 통해 팔로워의 뉴스 피드에 해당 포스트를 추가
   - 알림 서비스를 통해 팔로워에게 알림을 보냄
4. 데이터는 포스팅 캐시와 뉴스 피드 캐시에 저장되어 빠른 접근이 가능하도록 함
5. 최종적으로 데이터는 데이터베이스에 영구 저장된다

**피드 생성 흐름**

<!-- ![](images/p02-003.png) -->

1. 사용자가 `/v1/me/feed` 형식으로 피드를 요청
2. DNS와 로드밸런서를 거쳐 요청이 웹 서버로 전달
3. 웹 서버는 포스팅 전송 서비스를 통해 사용자의 친구들이 올린 포스트 데이터를 가져옴
4. 뉴스 피드 캐시에서 캐싱된 데이터를 확인하여 빠르게 피드를 생성
5. 최종적으로 생성된 피드가 사용자에게 전달

### 3단계: 상세 설계

이 단계로 왔다면 아래의 것을 달성한 것이다.

- 시스템에서 전반적으로 달성해야 할 목표와 기능 범위 확인
- 전체 설계의 개략적 청사진 마련
- 해당 청사진에 대한 면접관의 의견 청취
- 상세 설계서 집중해야 할 영역들 확인

**이제 면접관과 해야 할 일**: 설계 대상 컴포넌트 사이의 **우선순위**를 정하는 것.

> 대부분의 경우 면접관은 여러분이 특정 시스템 컴포넌트들의 세부사항을 깊이 있게 설명하는 것을 보길 원한다.

- **시간 관리**에 주의. 불필요한 세부사항에 시간을 쓰지 말라.
- 규모 확장 가능한 시스템을 설계할 능력이 있다는 것을 입증하는 데 집중해라.

#### 바뀐 구조에 대한 질문들

**Q. 왜 포스트 저장 서비스에는 메시지 큐를 사용하지 않나?**

> 포스팅 저장은 상대적으로 단순하고 빠른 작업이다. 데이터베이스와 캐시에 데이터를 쓰는 작업은 일반적으로 밀리초 단위로 완료되며, 시간이 오래 걸리는 작업이 아니다.
>
> 반면, 뉴스 피드 업데이트나 알림 전송은 수백, 수천 명의 친구에게 데이터를 전송해야 하므로 시간이 더 걸릴 수 있다. 이런 작업은 메시지 큐를 통해 비동기적으로 처리하는 것이 적합하다.

<!-- ![](images/p03-005.png) -->

**Q. 캐시를 3개나 두었는데 각각의 역할이 어떻게 되는가?**

> ⚠️ 그냥 나의 예상. 정확하진 X.

- **뉴스 피드 캐시**: 발행된 뉴스 피드를 가져온다.
- **포스팅 캐시**: 개별 포스트의 세부 정보(내용, 작성자, 작성 시간 등)를 캐싱.
  - 뉴스 피드 캐시가 "포스트 ID 목록"만 저장한다면, 포스팅 캐시는 해당 포스트 ID에 해당하는 실제 데이터를 제공.
  - 처음에 생각했던 방식인데 아닌 듯. 최신 뉴스피드를 가져오지 못하면 포스팅 캐시에서 가져오는 것 같음.
- **사용자 정보 캐시**: 뉴스 피드 캐시에 피드 데이터가 없거나 최신 데이터가 필요한 경우, 뉴스 피드 서비스는 사용자의 친구 목록을 조회해야 한다. 그때 사용.

### 4단계: 마무리

> 면접관이 시스템 병목 구간, 혹은 좀 더 개선 가능한 지점을 찾아내라 주문할 수 있다. 거기에 대고 여러분의 설계가 완벽했다거나 개선할 부분이 없다는 답은 하지 말자. 개선할 점은 언제나 있기 마련이며 이러한 질문은 비판적 사고 능력과 좋은 인상을 남길 기회이다.

**마무리에서 다룰 만한 것들**

- 설계를 다시 한번 요약. 여러 해결책을 제시한 경우 특히 중요.
- **오류 시나리오**: 서버 오류, 네트워크 장애 등이 발생하면 무슨 일이 생기는지.
- **운영 이슈**: 메트릭은 어떻게 수집/모니터링? 로그는? 배포는?
- **미래의 규모 확장 요구**에 어떻게 대처할 것인지.
- 시간이 좀 남았다면 필요하지만 다루지 못했던 세부적 개선사항들을 제안.

## 해야 할 것 / 하지 말아야 할 것

### ✅ 해야 할 것
- 질문을 통해 확인하라. 스스로 내린 가정이 옳다 믿고 진행하지 말라.
- 문제의 요구사항을 이해하라.
- **정답이나 최선의 답안 같은 것은 없다.** 스타트업을 위한 설계안과 수백만 사용자를 지원해야 하는 중견 기업을 위한 설계안이 같을 리 없다.
- 면접관이 여러분의 사고 흐름을 이해할 수 있도록 하라. 면접관과 소통하라.
- 가능하다면 여러 해법을 함께 제시하라.
- 개략적 설계에 면접관이 동의하면 각 컴포넌트의 세부사항을 설명하기 시작하라. **가장 중요한 컴포넌트부터** 진행하라.
- 면접관의 아이디어를 이끌어 내라. 좋은 면접관은 팀원처럼 협력한다.
- 포기하지 마라.

### ❌ 하지 말아야 할 것
- 전형적인 면접 문제들에도 대비하지 않은 상태에서 면접장에 가지 마라.
- 요구사항이나 가정들을 분명히 하지 않은 상태에서 설계를 제시하지 말라.
- 처음부터 특정 컴포넌트의 세부사항을 너무 깊이 설명하지 말라. 개략적 설계를 마친 뒤에 세부사항으로 나아가라.
- 진행 중에 막혔다면, 힌트 청하기를 주저하지 말라.
- 소통을 주저하지 말라. **침묵 속에 설계를 진행하지 말라.**
- 설계안을 내놓는 순간 면접이 끝난다고 생각하지 말라. 면접관이 끝났다고 말하기 전까지는 끝난 것이 아니다. 의견을 일찍, 그리고 자주 구하라.
