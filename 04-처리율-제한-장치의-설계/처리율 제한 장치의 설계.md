# 처리율 제한 장치의 설계

> 네트워크 시스템에서 **처리율 제한 장치(rate limiter)** 는 클라이언트 또는 서비스가 보내는 트래픽의 처리율을 제어하기 위한 장치다.

HTTP를 예로 들면 이 장치는 특정 기간 내에 전송되는 클라이언트 요청 횟수를 제한한다. 임계치를 넘어서면 추가로 도달한 모든 호출은 처리가 중단된다.

## 처리율 제한 사례 및 장점

**사례**
- 사용자는 초당 2회 이상 새 글을 올릴 수 없다.
- 같은 IP 주소로는 하루에 10개 이상의 계정을 생성할 수 없다.
- 같은 디바이스로는 주당 5회 이상 리워드를 요청할 수 없다.

**장점**
- **DoS(Denial of Service) 공격**에 의한 자원 고갈을 방지
- **비용 절감** — 서버를 많이 두지 않아도 된다
- **서버 과부하 방지**

## 1단계: 문제 이해 및 설계 범위 확정

**확인할 질문들**
- 어떤 종류의 처리율 제한 장치인가? 클라이언트 측인가, 서버 측인가?
- 어떤 기준을 사용해서 API를 제어해야 할까? IP 주소? 사용자 ID?
- 시스템 규모는? 스타트업 정도인지, 대규모 시스템인지?
- 분산 환경에서도 동작해야 하는지?
- 독립된 환경에서 동작해야 하는지, 기존 애플리케이션 코드에 포함되어야 하는지?
- 사용자에게 처리율 제한 장치에 의해 걸러진 경우 알려야 하는지?

### 요구사항

- 설정된 처리율을 초과하는 요청은 **정확하게** 제한한다.
- 낮은 응답 시간.
- 가능한 한 적은 메모리 사용.
- **분산 처리율 제한**: 하나의 처리율 제한 장치를 여러 서버나 프로세스에서 공유할 수 있어야 한다.
- **예외 처리**: 요청이 제한되었을 때는 그 사실을 사용자에게 분명하게 보여주어야 한다.
- **높은 결함 감내성(fault tolerance)**: 제한 장치에 장애가 생기더라도 전체 시스템에 영향을 주어서는 안 된다.

#### 💡 분산 시스템에서의 문제

분산 시스템에서는 여러 서버(또는 프로세스)가 동시에 요청을 처리한다. 각 서버가 독립적으로 카운터를 관리하면, 전체 요청 수를 정확히 추적할 수 없다.

> 예: 사용자 A가 1분에 100번 요청 제한이 있을 때
> - 서버 1: 60번 처리
> - 서버 2: 50번 처리
>
> 각 서버는 독립적으로 카운터를 관리하므로, 서버 1은 "60번", 서버 2는 "50번"으로만 알고 있다. 하지만 실제로는 총 110번 요청이 발생하여 제한(100번)을 초과했다.

→ **제한 수를 분산된 서버들이 공유해야 함.**

### 처리율 제한 알고리즘

널리 알려진 알고리즘은 다음과 같다.

- 토큰 버킷 (token bucket)
- 누출 버킷 (leaky bucket)
- 고정 윈도 카운터 (fixed window counter)
- 이동 윈도 로그 (sliding window log)
- 이동 윈도 카운터 (sliding window counter)

#### 토큰 버킷

> 인터넷 기업들이 보편적으로 사용. **아마존**과 **스트라이프**가 API 요청 통제에 사용.

**동작 원리**
- 토큰 버킷은 지정된 용량을 가지는 컨테이너다.
- 이 버킷에는 사전 설정된 양의 토큰이 주기적으로 채워진다.
- 토큰이 꽉 차면 더 이상의 토큰은 추가되지 않음. 버킷이 가득 차면 추가로 공급된 토큰은 버려진다.

<img width="482" height="229" alt="image" src="https://github.com/user-attachments/assets/bc9137aa-7096-4c10-b170-445f23decaf8" />

> 그림 4-4 예제는 용량이 4인 버킷이다.

**요청 처리**
- 각 요청은 처리될 때마다 하나의 토큰을 사용한다.
- 요청이 도착하면 버킷에 충분한 토큰이 있는지 검사한다.
  - **충분한 토큰이 있는 경우**: 버킷에서 토큰 하나를 꺼낸 후 요청을 시스템에 전달.
  - **충분한 토큰이 없는 경우**: 해당 요청은 버려진다(dropped).

<img width="525" height="438" alt="image" src="https://github.com/user-attachments/assets/59f998ae-f6fe-41a9-b2c0-fba1a76e857f" />

> **Q. 요청이 버려지면 안 되지 않나? 적어도 클라이언트 측에 알려줘야 하지 않나?**
> → 목적에 따라 다름.

| 전략 | 적합한 시스템 |
|---|---|
| **요청 버리기** (기본 토큰 버킷) | 실시간성이 중요한 시스템 (게임 서버, 스트리밍). 지연보다 요청 거부가 나은 경우. 클라이언트가 재시도 로직을 잘 처리할 수 있을 때. |
| **대기열 추가** | 요청 손실이 용납되지 않는 시스템 (결제, 예약). 지연이 어느 정도 허용. 공정성이 중요한 경우. |

<img width="393" height="401" alt="image" src="https://github.com/user-attachments/assets/da14d76a-1327-4df7-878e-5e7ba29e0b93" />

**인자 (2개)**
- **버킷 크기**: 버킷에 담을 수 있는 토큰의 최대 개수.
- **토큰 공급률**: 초당 몇 개의 토큰이 버킷에 공급되는가.

**버킷은 몇 개나 사용해야 할까?**
- 통상적으로 API 엔드포인트마다 별도의 버킷.
- IP 주소별로 처리율 제한을 적용한다면 IP 주소마다 버킷을 하나씩.
- 시스템 처리율을 초당 10,000개 요청으로 제한하고 싶다면 모든 요청이 하나의 버킷을 공유.

| 장점 | 단점 |
|---|---|
| 구현이 쉽다 | 2개의 인자(버킷 크기, 토큰 공급률)를 적절하게 튜닝하는 게 어렵다 |
| 메모리 사용 효율적 | |
| 짧은 시간에 집중되는 트래픽도 처리 가능 | |

#### 누출 버킷

> 토큰 버킷 알고리즘과 비슷하나 **요청 처리율이 고정**되어 있다는 점이 다르다. **FIFO 큐**로 구현.

**동작**
- 요청이 도착하면 큐가 가득 차 있는지 본다.
- 큐가 가득 차 있는 경우에는 새 요청은 버린다.
- 지정된 시간마다 큐에서 요청을 꺼내어 처리한다.

##### 토큰 버킷 vs 누출 버킷

| | 토큰 버킷 | 누출 버킷 |
|---|---|---|
| **동작** | 토큰을 일정 속도로 채움. 요청이 들어오면 토큰 소모/거부 | 큐가 일정 속도로 "누출"되어 처리. 가득 차면 새 요청 버림 |
| **버스트(Burst)** | 허용 (토큰이 쌓여 있으면 짧은 시간에 많이 처리) | 불가 (출력 속도 고정) |

<img width="528" height="229" alt="image" src="https://github.com/user-attachments/assets/82167302-b644-4380-be06-0718740e1b15" />

**인자 (2개)**
- **버킷 크기**: 큐 사이즈와 같은 값. 큐에는 처리될 항목들이 보관됨.
- **처리율 (outflow rate)**: 지정된 시간당 몇 개의 항목을 처리할지. 보통 초 단위.

> **쇼피파이(Shopify)** 가 이 알고리즘을 사용해서 처리율을 제한했다.

| 장점 | 단점 |
|---|---|
| 큐의 크기가 제한되어 메모리 효율적 | 단시간에 많은 트래픽이 몰리면 큐에 오래된 요청이 쌓이고, 최신 요청이 버려질 수 있음 |
| 고정된 처리율 → 안정적 출력 필요 시 적합 | 인자 값 튜닝이 까다로움 |

#### 고정 윈도 카운터

**동작**
- 타임라인을 고정된 간격의 윈도로 나누고, 각 윈도마다 카운터를 붙인다.
- 요청이 접수될 때마다 카운터 값은 1씩 증가.
- 카운터 값이 사전 설정된 임계치에 도달하면 새로운 요청은 새 윈도가 열릴 때까지 버려진다.

<img width="478" height="280" alt="image" src="https://github.com/user-attachments/assets/f37b971a-082d-45d3-a96c-d7cca655d416" />

> 예: 타임라인 단위 1초, 시스템은 초당 3개까지의 요청만 허용. 매초 열리는 윈도에 3개 이상 밀려오면 초과분은 버려진다.

**가장 큰 문제** — 윈도 경계 부근에 트래픽이 집중될 경우.

<img width="428" height="227" alt="image" src="https://github.com/user-attachments/assets/4174575a-8ad1-4704-9487-db577dae87d9" />

> 그림 4-9: 분당 최대 5개의 요청만 허용하는 시스템.
> - 카운터는 매분마다 초기화.
> - 1분 전에 5개가 들어왔고, 1분이 되면 초기화. 그리고 5개가 연속적으로 들어오면 이 시스템은 사실상 요청을 10개 처리한 것이 된다 → **허용 한도의 2배.**

| 장점 | 단점 |
|---|---|
| 메모리 효율적 | 윈도 경계 부근에 트래픽 집중 시 한도의 최대 2배까지 처리될 수 있음 |
| 이해하기 쉬움 | |
| 윈도가 닫히는 시점에 카운터를 초기화하는 방식은 특정 트래픽 패턴에 적합 | |

#### 이동 윈도 로깅

> 고정 윈도 카운터 알고리즘의 문제(경계 트래픽 집중)를 해결.

**동작**
- 요청의 **타임스탬프**를 추적한다. 보통 **레디스의 정렬 집합(sorted set)** 같은 캐시에 보관.
- 새 요청이 오면 만료된 타임스탬프는 제거. 만료된 타임스탬프 = 현재 윈도의 시작 시점보다 오래된 타임스탬프.
- 새 요청의 타임스탬프를 로그에 추가.
- 로그의 크기가 허용치보다 작거나 같으면 요청을 시스템에 전달. 아니면 거부.

##### 구체적 사례

<img width="513" height="342" alt="image" src="https://github.com/user-attachments/assets/a4757a09-a529-441d-95ef-50c6080a73f0" />

> 분당 최대 2회의 요청만을 처리하도록 설계.

| 시각 | 상태 |
|---|---|
| 1:00:01 | 로그가 비어 있음 → 허용 |
| 1:00:30 | [0:59:30, 1:00:30] 범위 내 요청 2개 미만 → 허용 |
| 1:00:50 | [0:59:50, 1:00:50] 범위 내 요청이 2개 초과 → **거부** (로그에는 남김) |
| 1:01:40 | [1:00:40, 1:01:40] 범위 내 요청 2개 미만 → 허용 (1:00:40 이전 로그는 지움) |

| 장점 | 단점 |
|---|---|
| 정교함 — 어느 순간의 윈도를 보더라도 허용된 요청 개수가 처리율 한도를 넘지 않음 | 다량의 메모리 사용 (거부된 요청의 타임스탬프도 보관하기 때문) |

> **Q. 거부된 요청도 타임스탬프에 보관하는 이유?**
> → 거부된 요청도 어쨌든 요청임.
>
> 예: 1:00:50에 거부된 요청을 로그에 남기지 않으면, 1:01:00에 새 요청이 들어왔을 때 윈도([1:00:00, 1:01:00])에 1:00:01과 1:00:30만 카운트되어 추가 요청이 허용될 수 있다. 제한(2회)을 초과하는 결과를 낳을 수 있음.

> 고정 윈도 카운터와 이동 윈도 로깅을 **결합**한 것.

<img width="511" height="275" alt="image" src="https://github.com/user-attachments/assets/aff65d3d-20e1-4fab-9dd3-127dd24740aa" />

**예시 — 분당 7개 처리율**
- 이전 1분 동안: 5개 요청
- 현재 1분 동안: 3개 요청
- 현재 1분의 30% 시점에 도착한 경우, 현재 윈도의 요청 수는?

> **공식**: 현재 1분간 요청 수 + 직전 1분간의 요청 수 × (이동 윈도와 직전 1분이 겹치는 비율)
>
> = 3 + 5 × 70% = **6.5** → 내림하여 6
>
> 분당 7개 요청이 한도이므로 신규 요청은 시스템으로 전달.

| 장점 | 단점 |
|---|---|
| 이전 시간대의 평균 처리율에 따라 현재 윈도 상태를 계산 → 짧은 시간에 몰리는 트래픽에도 잘 대응 | 직전 시간대 요청이 균등 분포라고 가정하여 추정 → 다소 느슨 |
| 메모리 효율적 | (실제 실험: 느슨해서 문제 생긴 데이터는 0.003% 정도) |

---

### 개략적인 아키텍처

**Q. 요청을 카운트하는 카운터는 어디에 보관할 것인가?**
- DB는 느려서 안 된다. **메모리 상에서 동작하는 캐시**가 바람직.
- 일례로 **Redis**는 처리율 제한 장치 구현에 자주 사용되며 다음 2가지 명령어 지원:
  - `INCR`: 메모리에 저장된 카운터 값을 1만큼 증가
  - `EXPIRE`: 카운터에 타임아웃 값을 설정. 설정된 시간이 지나면 자동 삭제

<img width="513" height="241" alt="image" src="https://github.com/user-attachments/assets/eaa95f5d-31eb-4392-afd1-0a8103a73e53" />

1. 클라이언트가 처리율 제한 미들웨어에게 요청을 보낸다.
2. 미들웨어는 레디스의 지정 버킷에서 카운터를 가져와 한도에 도달했는지 검사.
   - 한도 도달 → 요청 거부
   - 미달 → 요청은 API 서버로 전달

## 3단계: 상세 설계

- 분산 환경에서는 어떻게 처리할 것인가?

### 처리율 제한 규칙

### 처리율 한도 초과 트래픽의 처리

> 요청이 한도 제한에 걸리면 API는 **HTTP 429 (Too Many Requests)** 응답을 클라이언트에게 보낸다.

> 그냥 마무리 지을 수 있지만 나중에 처리하기 위해 **큐에 보관**할 수도 있다.

**클라이언트가 처리율 제한에 걸리고 있는지를 어떻게 감지?** → HTTP 응답 헤더에 정보를 담아주면 좋다.

| 헤더 | 의미 |
|---|---|
| `X-Ratelimit-Remaining` | 윈도 내에 남은 처리 가능 요청 수 |
| `X-Ratelimit-Limit` | 매 윈도마다 클라이언트가 전송할 수 있는 요청 수 |
| `X-Ratelimit-Retry-After` | 한도 제한에 걸리지 않으려면 몇 초 뒤에 요청을 다시 보내야 하는지 |

> 사용자가 너무 많은 요청을 보내면 429 오류를 `X-Ratelimit-Retry-After` 헤더와 함께 반환하자.

### 상세 설계

<img width="500" height="469" alt="image" src="https://github.com/user-attachments/assets/5c8d696a-475f-430e-aebe-9d6cf9dbb0e4" />

1. 처리율 제한 규칙은 **디스크**에 보관. 작업 프로세스(workers)는 수시로 규칙을 디스크에서 읽어 **캐시**에 저장.
2. 클라이언트가 요청을 서버에 보내면 요청은 먼저 처리율 제한 미들웨어에 도달.
3. 미들웨어는:
   - 제한 규칙을 캐시에서 가져옴.
   - 카운터 및 마지막 요청의 타임스탬프를 **Redis 캐시**에서 가져옴.
4. 결정:
   - **처리율 제한 미적용** → API 서버로 전달
   - **처리율 제한 적용** → 429 응답. 요청은 그대로 버리거나 메시지 큐에 보관할 수도 있음.

### 분산 환경에서의 처리율 제한

여러 서버를 지원하는 처리율 제한 장치를 구현하려면 다음 2가지 문제를 풀어야 한다.

- 경쟁 조건 (race condition)
- 동기화

#### 경쟁 조건

처리율 제한 장치는 대략 다음과 같이 동작.

1. 레디스에서 카운터 값을 읽는다.
2. `counter + 1`의 값이 임계치를 넘는지 본다.
3. 넘지 않는다면 레디스에 보관된 카운터 값을 1만큼 증가시킨다.

> 병행성이 심한 환경에서는 그림 4-14와 같은 경쟁 조건 이슈가 발생할 수 있다.

<img width="491" height="274" alt="image" src="https://github.com/user-attachments/assets/f72391ba-2087-464c-b256-db9c239d986b" />

**해결책**
- 일반적: **락(lock)** — 그러나 성능이 크게 떨어진다는 문제.
- 추천 2가지:
  - **루아 스크립트** (Lua script)
  - **정렬 집합** (sorted set) — Redis 자료구조

#### 동기화

- **고정 세션** 활용 가능 — but 확장에 유연하지 않으므로 추천 X
- **Redis 같은 중앙 집중형 데이터 저장소** 사용 추천

### 경성(soft) 또는 연성(hard) 처리율 제한
<!-- TODO: 영문 라벨이 반대. 경성=hard(절대 못넘음), 연성=soft(잠시 넘어도 됨). 책 원문도 hard/soft 매핑이 그러함. -->

- **경성 처리율 제한**: 요청의 개수는 임계치를 절대 넘어설 수 없다.
- **연성 처리율 제한**: 요청의 개수는 잠시 동안은 임계치를 넘어설 수 있다.

### 다양한 계층에서의 처리율 제한

> 지금까지의 설명은 OSI 7계층의 **애플리케이션 계층**을 기준. 다른 계층에서도 처리율 제한 가능.
> ex. OSI 3번 계층, `iptables`를 사용하여 IP 주소에 처리율 제한을 적용할 수 있음.

**참고 — OSI 계층**

| 계층 | 이름 |

| 1 | 물리 |
| 2 | 데이터 링크 |
| 3 | 네트워크 |
| 4 | 전송 |

| 6 | 표현 (Presentation) |

### 처리율 제한을 회피하는 방법 — 클라이언트 측 설계

- 클라이언트 측 **캐시**를 사용하여 API 호출 횟수를 줄인다.
- 처리율 제한의 **임계치를 이해**하고, 짧은 시간 동안 너무 많은 메시지를 보내지 않도록 한다.
- 예외나 에러를 처리하는 코드를 도입하여 클라이언트가 예외적 상황으로부터 **우아하게 복구**될 수 있도록 한다.
- **재시도 로직**을 구현할 때는 충분한 **백오프(back-off)** 시간을 둔다.
