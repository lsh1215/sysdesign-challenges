### 서버 추가/제거 시의 문제

예를 들어 1번 서버가 고장났다고 하자. 서버 풀의 크기는 3으로 변하고 나머지 연산을 적용해서 계산한 서버 인덱스 값이 달라질 것이다.

<!-- ![](images/p02-005.png) -->

<!-- ![](images/p02-006.png) -->

> 그림 5-2에서 보인 대로 장애가 발생한 1번 서버에 보관되어 있는 키뿐만 아닌 **대부분의 키가 재분배**되었다.
>
> → 1번 서버가 죽으면 대부분의 캐시 클라이언트가 데이터가 없는 엉뚱한 서버에 접속하게 된다는 뜻. 그 결과로 **대규모 캐시 미스**가 발생.

**안정 해시**는 이 문제를 효과적으로 해결하는 기술이다.

## 안정 해시

> **정의**: 해시 테이블 크기가 조정될 때 평균적으로 오직 **k/n 개의 키**만 재배치하는 해시 기술. 여기서 k는 키의 개수, n은 슬롯의 개수.
>
> 이와는 달리 대부분 전통적 해시 테이블은 슬롯의 수가 바뀌면 거의 대부분 키를 재배치한다.

### 해시 공간과 해시 링

> SHA-1을 사용한다는 가정하에 설명을 진행한다.

- 해시 함수의 출력 값 범위를 `x0` 부터 `xn`이라고 하자.
- SHA-1의 **해시 공간(hash space)** 범위는 `0` 부터 `2^160 - 1` 까지.
- 따라서 `x0`는 0, `xn`은 `2^160 - 1`이며, 나머지는 그 사이 값.

<!-- ![](images/p02-007.png) -->

이 해시 공간을 양쪽으로 구부려 **해시 링**을 만들어보자.

<!-- ![](images/p02-008.png) -->

### 해시 서버 / 해시 키

해시 함수 `f`를 이용하면 서버 IP나 이름을 링 위의 어떤 위치에 대응시킬 수 있다.

<!-- ![](images/p03-009.png) -->

> **해시 키 함수**는 "해시 키 재배치 문제"에 언급된 함수와는 다르다. **나머지 연산 `%`을 사용하지 않는다.**

<!-- ![](images/p03-010.png) -->

### 서버 조회

> 어떤 키가 저장되는 서버는, 해당 키의 위치로부터 **시계 방향으로** 링을 탐색해 나가면서 만나는 **첫 번째 서버**다.

<!-- ![](images/p03-011.png) -->

### 서버 추가 / 제거

#### 서버 추가

안정 해시에서는 서버를 추가하더라도 **키 가운데 일부만** 재배치하면 된다.

> 예: `s0`와 `s3` 사이에 `s4`라는 새로운 서버가 추가되었을 때, `s0`와 `s4` 사이에 있는 키들(여기서는 `k0`)만 `s4`로 재배치하면 된다.

<!-- ![](images/p04-012.png) -->

#### 서버 제거

마찬가지로 서버를 제거하더라도 일부만 재배치.

> 예: `s1` 서버가 제거되었을 때, `s1`에 할당되어 있던 키들(`s0`와 `s1` 사이의 키들, 여기서는 `k1`)만 `s2`로 재배치하면 된다.

<!-- ![](images/p04-013.png) -->

## 기본 구현법의 2가지 문제

> 안정 해시는 MIT에서 처음 제안되었다.

**기본 절차**
1. 서버와 키를 균등 분포 해시 함수를 사용해 해시 링에 재배치한다.
2. 키의 위치에서 링을 시계 방향으로 탐색하다 만나는 최초의 서버가 키가 저장될 서버다.

### 문제 1: 파티션 크기 불균등

> **파티션**: 인접한 서버 사이의 해시 공간.

서버가 추가되거나 삭제되는 상황을 감안하면, 파티션 크기를 균등하게 유지하는 게 불가능하다.

> 위의 "서버 삭제" 예시에서는 `s1`이 삭제되었기 때문에 `s2`의 파티션이 다른 파티션 대비 거의 두 배로 커진다.

<!-- ![](images/p05-014.png) -->

### 문제 2: 키 분포 불균등

해시 값에 따라 배치가 되기 때문에 균등 분포가 달성되기 어려울 수도 있다.

> 예: 서버 1과 서버 3은 아무 데이터를 갖지 않는 반면 대부분의 키는 서버 2에 보관될 수 있다.

<!-- ![](images/p05-015.png) -->

## 가상 노드

> 위의 2가지 문제를 해결하기 위해 **가상 노드(virtual node)** 개념을 도입.

- 가상 노드는 실제 노드 또는 서버를 가리키는 노드.
- 하나의 서버는 링 위에 여러 개의 가상 노드를 가질 수 있다.

> 예: 서버 0와 서버 1은 3개의 가상 노드를 가진다. (여기서 숫자 3은 임의로 정한 것, 실제로는 훨씬 큰 값.)
> - `s0` 하나만 쓰는 대신 `s0_0`, `s0_1`, `s0_2`
> - `s1`도 마찬가지

<!-- ![](images/p05-016.png) -->

<!-- ![](images/p06-017.png) -->

### 가상 노드 개수의 영향

- 가상 노드의 개수를 늘리면 키의 분포는 점점 균등해진다. (표준 편차가 작아짐)
- **100 ~ 200개**의 가상 노드를 사용했을 경우 표준 편차의 값은 **평균 5% ~ 10% 사이**.
- 가상 노드를 무한정 늘리면 파티션이 더 균등하게 배분되지만 **노드를 저장할 공간이 더 많이 필요**하므로 타협이 필요.

### 서버 추가 시

> `s4`가 추가된 경우, 영향을 받는 범위는 `s4`부터 **반시계 방향**에 있는 첫 번째 서버 `s3`까지. → `s3`부터 `s4` 사이의 키들을 `s4`로 재배치.

## 마치며

### 안정 해시의 이점

- 서버가 추가되거나 삭제될 때 재배치되는 키의 수가 **최소화**된다.
- 데이터가 보다 균등하게 분포 → 수평적 규모 확장성 달성이 쉽다.
- **핫스팟(hotspot) 키** 문제를 줄인다.
  - 특정한 샤드(shard)에 대한 접근이 지나치게 빈번하면 서버 과부하 문제가 생길 수 있다. 안정 해시는 데이터를 좀 더 균등하게 분배하므로 이런 문제 가능성을 줄인다.

### 안정 해시가 쓰이는 곳의 예시

- 아마존 **다이나모(DynamoDB)** 의 파티셔닝 관련 컴포넌트
- 아파치 **카산드라(Cassandra)** 클러스터에서의 데이터 파티셔닝
- **디스코드(Discord)** 채팅 애플리케이션
- **아카마이(Akamai)** CDN
