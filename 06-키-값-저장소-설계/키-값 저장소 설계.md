# 키-값 저장소 설계

> **키-값 저장소(key-value store)** 는 키-값 데이터베이스라고도 불리는 **비관계형(non-relational) 데이터베이스**.
> 저장되는 값은 **고유 식별자**를 가져야 한다. 키와 값 사이의 이런 연결 관계를 "키-값 쌍(pair)"이라고 지칭한다.

## 기본 개념

- 키는 **유일**해야 하며 해당 키에 매달린 값은 키를 통해서만 접근 가능.
- 키는 일반 텍스트일 수도 있고, 해시 값일 수도 있다.
- 성능상의 이유로 키는 **짧을수록 좋다**.

**널리 알려진 키-값 저장소**
- 아마존 다이나모
- memcached
- redis

**이번 장에서 설계할 키-값 저장소의 연산**

| API | 설명 |
|---|---|
| `put(key, value)` | 키-값 쌍을 저장소에 저장 |
| `get(key)` | 인자로 주어진 키에 매달린 값을 꺼냄 |

## 문제 이해 및 설계 범위 확정

> **완벽한 설계란 없다.**
> 읽기, 쓰기, 그리고 메모리 사용량 사이에 어떤 균형을 찾고, 데이터 일관성과 가용성 사이에서 타협적 결정을 내린 설계를 만들었다면 쓸 만한 답안일 것이다.

**요구사항**
- 키-값 쌍의 크기는 **10KB 이하**
- 큰 데이터를 저장할 수 있어야 한다
- **높은 가용성** — 설사 장애가 있더라도 빨리 응답해야 한다
- **높은 규모 확장성** — 트래픽 양에 따라 자동적으로 서버 증설/삭제
- **데이터 일관성 수준은 조정이 가능**해야 한다
- **응답 지연 시간(latency)** 이 짧아야 한다

## 단일 서버 키-값 저장소

> 한 대의 서버만 사용하는 키-값 저장소를 설계하는 것은 쉽다.
> 가장 직관적인 방법: 키-값 쌍 전부를 **메모리에 해시 테이블**로 저장.

**약점**: 모든 데이터를 메모리 안에 두는 것이 불가능할 수도 있다.

**개선책**
- 데이터 압축(compression)
- 자주 쓰는 데이터만 메모리에 두고 나머지는 디스크에 저장

> 그러나 이렇게 개선해도 한 대 서버로 부족한 때가 곧 찾아온다. → **분산 키-값 저장소**가 필요.

## 분산 키-값 저장소

> 분산 키-값 저장소는 **분산 해시 테이블**이라고도 불린다 — 키-값 쌍을 여러 서버에 분산시키는 탓이다.

이러한 분산 시스템을 설계할 때는 **CAP 정리**를 이해하고 있어야 한다.

### CAP 정리

> CAP 정리: 다음 세 가지를 **동시에** 만족하는 분산 시스템을 설계하는 것은 불가능하다.

| 약자 | 이름 | 의미 |
|---|---|---|
| **C** | 데이터 일관성 (Consistency) | 분산 시스템에 접속하는 모든 클라이언트는 어떤 노드에 접속했느냐에 관계없이 언제나 같은 데이터를 보게 되어야 한다 |
| **A** | 가용성 (Availability) | 분산 시스템에 접속하는 클라이언트는 일부 노드에 장애가 발생하더라도 항상 응답을 받을 수 있어야 한다 |
| **P** | 파티션 감내 (Partition Tolerance) | 파티션 = 두 노드 사이에 통신 장애가 발생하였음. 파티션 감내 = 네트워크에 파티션이 생기더라도 시스템은 계속 동작해야 한다 |

> 포통 파티션 감내는 필수적이다.

> CAP 정리는 이들 가운데 어떤 두 가지를 충족하려면 나머지 하나는 **반드시 희생**되어야 한다는 것을 의미한다.

#### CP / AP / CA 시스템

- **CP 시스템**
  - 일관성과 파티션 감내를 지원 — **가용성을 희생**.
  - 네트워크가 끊기면 일부 노드가 응답 안 할 수 있음.
- **AP 시스템**
  - 일관성과 파티션 감내를 지원하는 키-값 저장소
    <!-- TODO: AP는 "**가용성**(Availability)과 파티션 감내"를 지원, 일관성을 희생함. 첫 줄이 CP 설명을 그대로 복붙한 듯. -->
  - **데이터 일관성을 희생**.
  - 네트워크 끊겨도 응답은 하지만 데이터가 최신이 아닐 수 있음.
- **CA 시스템**
  - 일관성과 가용성을 지원 — 파티션 감내를 희생.
  - **실세계에 CA 시스템은 없다.**

#### 이상적 상태

> 이상적 환경이라면 네트워크가 파티션되는 상황은 존재하지 않을 것이다. `n1`에 기록된 데이터는 자동적으로 `n2`와 `n3`에 복제된다. 데이터 일관성과 가용성도 만족.

<img width="684" height="586" alt="image" src="https://github.com/user-attachments/assets/32084d67-3cb1-42ca-b118-83e184ef1700" />

#### 실세계의 분산 시스템

> 분산 시스템은 파티션 문제를 피할 수 없다. 파티션 문제가 발생하면 우리는 **일관성과 가용성 사이에서 하나를 선택**해야 한다.

> 그림은 `n3`에 장애가 발생해서 `n1`과 `n2`와 통신할 수 없는 상황. `n3`에 기록되었으나 아직 `n1`/`n2`로 전달되지 않은 데이터가 있다면 `n1`/`n2`는 오래된 사본을 갖고 있을 것.

**일관성을 선택한 경우 (CP)**
- 가용성 대신 일관성을 선택한다면 데이터 불일치를 피하기 위해 `n1`과 `n2`에 대해 쓰기 연산을 중단시켜야 함 → **가용성이 깨진다.**
- **은행권**이 대표적인 CP 시스템 (데이터 일관성이 중요).
- 네트워크 파티션 때문에 일관성이 깨질 수 있는 상황이 발생하면, 이런 시스템은 상황이 해결될 때까지 **오류를 반환**해야 한다.

**가용성을 선택한 경우 (AP)**
- 일관성 대신 가용성을 선택한 시스템은 설사 낡은 데이터를 반환하더라도 계속 읽기 연산을 허용.
- `n1`과 `n2`는 계속 쓰기 연산을 허용하고, 파티션 문제가 해결된 뒤에 새 데이터를 `n3`에 전송.

### 데이터 파티션

> 대규모 애플리케이션의 경우 전체 데이터를 한 대 서버에 욱여넣는 것은 불가능. 가장 단순한 해결책: **데이터를 작은 파티션으로 분할하고 여러 대의 서버에 저장**.

**파티션 분할 시 중요하게 따질 2가지**
1. 데이터를 여러 서버에 **고르게 분산**할 수 있는가?
2. 노드가 추가되거나 삭제될 때 **데이터의 이동을 최소화**할 수 있는가?

> **안정 해시(consistent hash)** 가 이 문제를 푸는 데 적합한 기술이다.

**안정 해시로 파티션하면 좋은 점**
- **규모 확장 자동화**: 시스템 부하에 따라 서버가 자동으로 추가되거나 삭제되도록 만들 수 있다.
- **다양성(heterogeneity)**: 각 서버의 용량에 맞게 가상 노드의 수를 조정할 수 있다. 고성능 서버는 더 많은 가상 노드.

---

### 데이터 다중화

> 높은 가용성과 안정성을 확보하기 위해 데이터를 **N개 서버에 비동기적으로 다중화(replication)**. 여기서 N은 튜닝 가능한 값.

<img width="1242" height="1040" alt="image" src="https://github.com/user-attachments/assets/d9942e1a-cad7-40e0-8ff1-d7d128867d9d" />

> 가상 노드를 사용할 경우 위와 같이 N개의 노드가 대응될 **실제 물리 서버의 개수가 N보다 작아질 수 있다.** 이 문제를 피하려면 노드를 선택할 때 같은 물리 서버를 중복 선택하지 않도록 해야 한다.

#### 풀어서 이해하기

**상황**: 데이터를 N=3으로 복제. 시스템에는 물리 서버 2대만 있다고 가정.

- **가상 노드 없이**
  - 물리 서버 2대가 각각 1개의 노드라면, N=3을 만족할 수 없다 → 복제 불가/제한.
- **가상 노드 사용**
  - 서버 A를 2개의 가상 노드(A1, A2), 서버 B를 1개의 가상 노드(B1)로 설정.
  - 총 3개의 가상 노드 → N=3 만족 가능.

→ 즉, **가상 노드 덕분에 물리 서버 개수가 N보다 작아도 N개 복제 가능.**

#### 데이터 센터 분리

> 같은 데이터 센터에 속한 노드는 문제를 동시에 겪을 가능성이 있다. 안정성 담보를 위해 데이터의 사본은 **다른 센터의 서버**에 보관하고, 센터들은 **고속 네트워크**로 처리한다.

---

### 데이터 일관성

> 여러 노드에 다중화된 데이터는 적절히 동기화되어야 한다. **정족수 합의 프로토콜(quorum consensus)** 을 사용하면 읽기/쓰기 연산 모두에 일관성을 보장할 수 있다.

> **정족수**: 여러 사람의 합의로 운영되는 의사기관에서 의결을 하는 데 필요한 최소한의 참석자 수.

**기호 정의**

| 기호 | 의미 |
|---|---|
| `N` | 사본 개수 |
| `W` | 쓰기 연산에 대한 정족수 — 쓰기가 성공한 것으로 간주되려면 적어도 W개의 서버로부터 성공 응답 |
| `R` | 읽기 연산에 대한 정족수 — 읽기가 성공한 것으로 간주되려면 적어도 R개의 서버로부터 응답 |

#### 예시: N=3

<img width="868" height="688" alt="image" src="https://github.com/user-attachments/assets/479f994e-cf8f-4eed-acc9-e1cba3b4bbd8" />

> `W = 1`은 데이터가 한 대 서버에만 기록된다는 뜻이 아니다.
>
> 그림 6-6과 같이 데이터가 `s0, s1, s2`에 다중화되는 상황. `W = 1`의 의미는 쓰기가 성공했다고 판단하기 위해 **중재자가 최소 한 대 서버로부터 성공 응답을 받아야 한다는 뜻**이다.
>
> 따라서 `s1`으로부터 성공 응답을 받았다면 `s0, s2`로부터 응답은 기다릴 필요 없다.

> **중재자**는 클라이언트와 노드 사이에서 **프록시 역할**을 한다.

#### N, W, R 값을 정하는 가이드

> `W, R, N`의 값을 정하는 것은 **응답 지연과 데이터 일관성 사이 타협점**을 찾는 전형적인 과정.

- `W = 1` 또는 `R = 1`: 중재자가 한 대 서버 응답만 받으면 되니 **응답 속도 빠름**.
- `W`나 `R`이 1보다 크면: **일관성 향상**, but 응답 속도는 가장 느린 서버를 기다려야 하므로 느려짐.

| 설정 | 특성 |
|---|---|
| `R = 1, W = N` | 빠른 읽기에 최적화 |
| `W = 1, R = N` | 빠른 쓰기에 최적화 |
| `W + R > N` | 강한 일관성 보장 (보통 N=3, W=R=2) |
| `W + R ≤ N` | 강한 일관성 보장되지 않음 |

### 일관성 모델

> 일관성 모델은 키-값 저장소를 설계할 때 고려해야 할 중요한 요소 중 하나. 데이터 일관성의 수준을 결정한다.

| 모델 | 설명 |
|---|---|
| **강한 일관성** | 모든 읽기 연산은 가장 최근에 갱신된 결과를 반환. 절대로 낡은 데이터를 보지 못한다 |
| **약한 일관성** | 읽기 연산은 가장 최근에 갱신된 결과를 반환하지 못할 수 있다 |
| **최종 일관성** | 약한 일관성의 한 형태. 갱신 결과가 결국에는 모든 사본에 반영되는 모델 |

**강한 일관성을 달성하는 방법**
- 모든 사본에 현재 쓰기 연산의 결과가 반영될 때까지 데이터에 대한 읽기/쓰기를 금지.
- 이 방법은 **고가용성 시스템에는 적합하지 않다** (새로운 요청의 처리가 중단됨).

> **다이나모** 또는 **카산드라** 같은 저장소는 **최종 일관성 모델**을 택하고 있다. 여기서도 최종 일관성 모델을 기준으로 키-값 저장소를 설계한다.

> 최종 일관성 모델을 따를 경우 쓰기 연산이 병렬적으로 발생하면 시스템에 저장된 값의 일관성이 깨질 수 있는데, 이 문제는 **클라이언트가 해결**해야 한다.

### 비일관성 해소 기법: 데이터 버저닝

> 데이터를 다중화 하면 가용성은 높아지지만 사본 간 일관성이 깨질 가능성은 높아진다.
> **버저닝(versioning)** 과 **벡터 시계(vector clock)** 는 이 문제를 해소하기 위해 등장한 기술이다.

- **버저닝**: 데이터를 변경할 때마다 해당 데이터의 새로운 버전을 만든다. 각 버전의 데이터는 변경 불가능.

#### 일관성이 깨지는 예시

> 어떤 데이터의 사본이 노드 `n1`과 `n2`에 보관 중. 서버 1과 서버 2가 `get("name")`으로 같은 값을 얻는다.

<img width="872" height="514" alt="image" src="https://github.com/user-attachments/assets/6d667ad1-c49f-45ac-806b-5c7b933b9277" />

> 서버 1은 "name"을 `johnSanFrancisco`로, 서버 2는 `johnNewYork`으로 동시에 바꾼다. → 두 충돌하는 값 `v1`, `v2`.

<img width="996" height="502" alt="image" src="https://github.com/user-attachments/assets/65a8c66d-0db5-4608-9159-5b35d23e54f8" />

> 변경 이후 원래 값은 무시 가능. 하지만 마지막 두 버전 `v1`과 `v2` 사이의 충돌은 해소하기 어려워 보인다.
> → 충돌을 발견하고 자동으로 해결할 **버저닝 시스템**이 필요. **벡터 시계**는 이런 문제를 푸는 데 보편적으로 사용된다.

#### 벡터 시계

> **벡터 시계** = `[서버, 버전]`의 순서쌍을 데이터에 매단 것. 어떤 버전이 선행/후행 버전인지, 다른 버전과 충돌이 있는지 판별하는 데 쓰인다.

벡터 시계는 `D([S1, v1], [S2, v2], …, [Sn, vn])`와 같이 표현. `D`는 데이터, `vi`는 버전 카운터, `Si`는 서버 번호.

데이터 `D`를 서버 `Si`에 기록하면 시스템은 다음 작업 중 하나를 수행:
- `[Si, vi]`가 있으면 `vi`를 증가시킨다.
- 그렇지 않으면 새 항목 `[Si, 1]`을 만든다.

<img width="802" height="770" alt="image" src="https://github.com/user-attachments/assets/4cdba460-8174-42d7-ac15-7ee9ed721bdc" />

**시나리오 예시**

1. 클라이언트가 데이터 `D1`을 시스템에 기록. 쓰기 처리 서버는 `Sx`. → `D1([Sx, 1])`
2. 다른 클라이언트가 `D1`을 읽고 `D2`로 업데이트. `D1`을 덮어쓴다. 같은 서버 `Sx`가 처리. → `D2([Sx, 2])`
3. 다른 클라이언트가 `D2`를 읽어 `D3`로 갱신. `Sy`가 처리. → `D3([Sx, 2], [Sy, 1])`
4. 또 다른 클라이언트가 `D2`를 읽고 `D4`로 갱신. `Sz`가 처리. → `D4([Sx, 2], [Sz, 1])`
5. 어떤 클라이언트가 `D3`와 `D4`를 읽으면 데이터 간 **충돌**이 있다는 것을 알게 된다 (`D2`를 `Sy`와 `Sz`가 각기 다른 값으로 바꿈). 클라이언트가 충돌을 해소한 후 서버에 기록. `Sx`가 처리. → `D5([Sx, 3], [Sy, 1], [Sz, 1])`

#### 선후 관계 / 충돌 판별

- **선행 버전 판별**: 버전 Y에 포함된 모든 구성요소의 값이 X에 포함된 모든 구성요소 값보다 같거나 크면 X가 Y의 이전 버전.
  - 예: `D([s0,1], [s1,1])` 는 `D([s0,1], [s1,2])` 의 이전 버전 → **충돌 없음**.
- **충돌 판별**: Y의 벡터 시계 구성요소 가운데 X의 동일 서버 구성요소보다 **작은 값**을 갖는 것이 있는지 본다.
  - 예: `D([s0,1], [s1,2])` 와 `D([s0,2], [s1,1])` → **서로 충돌**.

#### 벡터 시계의 단점

1. 충돌 감지 및 해소 로직이 **클라이언트에 들어가야 하므로 클라이언트 구현이 복잡**해진다.
2. `[서버:버전]` 순서쌍 개수가 굉장히 빨리 늘어난다.
   - 해결: 길이에 임계치를 설정하고, 임계치 이상으로 길어지면 오래된 순서쌍을 제거.
   - 이론적으로는 버전 간 선후 관계가 정확하게 결정될 수 없어 충돌 해소 효율성이 낮아지지만, **아마존의 문헌에 따르면 실제 서비스에서 그런 문제가 벌어지는 것을 발견한 적이 없다**고 한다.

### 장애 처리

> 대다수 대규모 시스템에서 장애는 그저 불가피하기만 한 것이 아니라 **아주 흔하다.**

#### 장애 감지

> 분산 시스템에서는 그저 한 대 서버가 "지금 서버 A가 죽었습니다"라고 한다 해서 바로 서버 A를 장애 처리하지는 않는다. 보통 **두 대 이상의 서버가 똑같이** 서버 A의 장애를 보고해야 한다.

**방법 1: 모든 노드 사이 멀티캐스팅 채널**
- 가장 손쉬운 방법.
- but 서버가 많을 때 비효율적.

<img width="632" height="538" alt="image" src="https://github.com/user-attachments/assets/76169fdf-905c-4316-8e93-0caf7447805b" />

**방법 2: 가십 프로토콜 (Gossip Protocol)** — 추천

> 분산형 장애 감지 솔루션. 보다 효율적.

**동작 원리**
- 각 노드는 **멤버십 목록**을 유지. (각 멤버 ID와 그 박동 카운터 쌍)
- 각 노드는 주기적으로 자신의 박동 카운터를 증가시킨다.
- 각 노드는 무작위로 선정된 노드들에게 주기적으로 자기 박동 카운터 목록을 보낸다.
- 박동 카운터 목록을 받은 노드는 멤버십 목록을 최신 값으로 갱신.
- 어떤 멤버의 박동 카운터 값이 지정된 시간 동안 갱신되지 않으면 해당 멤버는 **장애 상태**로 간주.

<img width="1010" height="516" alt="image" src="https://github.com/user-attachments/assets/66a20513-a095-474e-bba2-e49ec8bd7276" />

> - 노드 `s0`는 멤버십 목록을 가진 상태.
> - 노드 `s0`는 노드 `s2`의 박동 카운터가 오랫동안 증가되지 않았다는 것을 발견.
> - `s0`는 `s2`를 포함하는 박동 카운터 목록을 무작위로 선택된 다른 노드에게 전달.
> - `s2`의 박동 카운터가 오랫동안 증가되지 않았음을 발견한 모든 노드는 해당 노드를 **장애 노드로 표시**.

#### 일시적 장애 처리

가십 프로토콜로 장애를 감지한 시스템은 가용성을 보장하기 위해 필요한 조치를 해야 한다.

- **엄격한 정족수 접근법**: 읽기와 쓰기 연산을 금지.
- **느슨한 정족수 접근법** (sloppy quorum): 정족수 요구사항을 강제하는 대신, 쓰기 W개의 건강한 서버와 읽기 R개의 건강한 서버를 해시 링에서 고른다 (장애 서버는 무시).

> 장애 서버로 가는 요청은 다른 서버가 잠시 맡아 처리. 복구되었을 때 그동안 발생한 변경사항은 일괄 반영하여 데이터 일관성을 보존. 이를 위해 임시로 쓰기 연산을 처리한 서버에는 그에 관한 단서(hint)를 남겨둔다.
>
> → 이러한 장애 처리 방안을 **"단서 후 임시 위탁(hinted handoff)"** 기법이라 부른다.

<img width="678" height="648" alt="image" src="https://github.com/user-attachments/assets/2f6b368c-aa5d-433b-92dd-4e99edc81798" />

> 장애 상태인 노드 `s2`에 대한 읽기/쓰기 연산은 일시적으로 `s3`가 처리. `s2`가 복구되면 `s3`은 갱신된 데이터를 `s2`로 인계.

#### 영구 장애 처리

> 단서 후 임시 위탁 기법은 **일시적 장애** 처리용. **영구적 노드 장애**에는 → **반-엔트로피(anti-entropy) 프로토콜**.

- 반-엔트로피 프로토콜: 사본들의 데이터를 비교하여 각 사본을 최신 버전으로 갱신.
- 사본 간 일관성이 망가진 상태를 탐지하고 전송 데이터 양을 줄이기 위해 **머클(Merkle) 트리**를 사용하면 좋다.

##### 머클 트리란?

> **해시 트리(hash tree)** 라고도 불리는 머클 트리는 각 노드에 그 자식 노드들에 보관된 값의 해시(자식 노드가 종단 노드인 경우), 또는 자식 노드들의 레이블로부터 계산된 해시 값을 레이블로 붙여두는 트리. (wiki)

**키 공간 1~12 예제**

> 일관성이 망가진 데이터가 위치한 상자는 다른 색으로 표시.

1. 키 공간을 그림 6-13과 같이 버킷으로 나눈다. (본 예제는 4개 버킷)
   <img width="956" height="384" alt="image" src="https://github.com/user-attachments/assets/d2ebe66a-3b07-44dd-98c8-0e3336f82088" />

2. 버킷에 포함된 각각의 키에 균등 분포 해시 함수를 적용하여 해시 값을 계산.
   <img width="1000" height="380" alt="image" src="https://github.com/user-attachments/assets/e88fff12-0daa-4b1b-a548-7e148ed78c90" />

3. 버킷별로 해시값을 계산하고, 해당 해시 값을 레이블로 갖는 노드를 만든다.
   <img width="948" height="396" alt="image" src="https://github.com/user-attachments/assets/6bf5f3d7-f865-45f6-b0a5-9ee98cfd680e" />

4. 자식 노드의 레이블로부터 새로운 해시 값을 계산하여 이진 트리를 상향식으로 구성.
   <img width="928" height="544" alt="image" src="https://github.com/user-attachments/assets/69bb0762-c6e4-4d8e-8cab-3003396d7b65" />

**비교 방식**
- 두 머클 트리의 비교는 **루트 노드 해시값 비교**로 시작.
- 루트 해시 값이 일치 → 두 서버는 같은 데이터.
- 값이 다르면 왼쪽 자식, 오른쪽 자식 순으로 비교.
- 이렇게 하다 보면 다른 데이터를 갖는 **버킷을 찾을 수 있고, 그 버킷들만 동기화**하면 된다.

> 머클 트리를 사용하면 동기화해야 하는 데이터 양은 **실제로 존재하는 차이의 크기에 비례할 뿐 두 서버에 보관한 데이터 총량과는 무관.** (차이가 작으면 전송량도 작다.)
>
> 실제 시스템의 경우 버킷 하나의 크기가 꽤 크다. 예: 10억 개의 키를 100만 개의 버킷 → 버킷 하나당 1,000개의 키.

#### 데이터 센터 장애 처리

> 데이터 센터 장애는 다양한 이유로 발생할 수 있으니 **여러 데이터 센터에 데이터를 다중화** 해놓자.

### 쓰기 경로

> 쓰기 요청이 특정 노드에 전달되면 무슨 일이 벌어지는지 (카산드라 사례).

<img width="890" height="536" alt="image" src="https://github.com/user-attachments/assets/29c88142-9f62-4bac-8bd8-a90f25678802" />

1. 쓰기 요청이 **커밋 로그 파일**에 기록된다.
2. 데이터가 **메모리 캐시**에 기록된다.
3. 메모리 캐시가 가득 차거나 사전 정의된 임계치에 도달하면 데이터는 디스크에 있는 **SSTable**에 기록된다.
   - `SSTable` = **Sorted-String Table**. `<키, 값>` 순서쌍을 정렬된 리스트 형태로 관리하는 테이블.

### 읽기 경로

> 읽기 요청을 받은 노드는 **데이터가 메모리 캐시에 있는지부터** 살핀다.
> 있으면 그림 6-20과 같이 해당 데이터를 클라이언트에 반환.

데이터가 메모리에 없는 경우 디스크에서 가져와야 한다. 어느 SSTable에 찾는 키가 있는지 알아낼 효율적인 방법이 필요한데 **블룸 필터**가 흔히 사용된다.

<img width="1016" height="542" alt="image" src="https://github.com/user-attachments/assets/327b3404-96a9-4701-99de-1c950855db25" />

2. 데이터가 메모리에 없으므로 **블룸 필터 검사**.
3. 블룸 필터를 통해 어떤 SSTable에 키가 보관되어 있는지 찾는다.
4. SSTable에서 데이터를 가져온다.
5. 해당 데이터를 클라이언트에게 반환.
