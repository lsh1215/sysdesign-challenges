# 분산 시스템을 위한 유일 ID 생성기 설계

> **Q.** ID를 생성할 때 `auto_increment` 속성이 설정된 관계형 DB의 기본키를 사용하면 되지 않을까?
>
> **A.** 그러나 이러한 생각은 **분산 환경에서는 먹히지 않는다.**
>
> 여러 DB들이 중복이 일어나지 않게 id 값이 어디까지 저장되어 있는지 공유해야 될 텐데, 네트워크 지연이라도 돼서 id 공유가 업데이트보다 느리게 된다면 중복되는 id들이 생길 것이다.
>
> → **처음부터 중복되지 않는 ID를 만들면 된다.**

## 1단계: 문제 이해 및 설계 범위 확정

**요구사항**
- ID는 **유일**해야 한다.
- ID는 **숫자로만** 구성되어야 한다.
- ID는 **64비트**로 표현될 수 있는 값이어야 한다.
- ID는 **발급 날짜에 따라 정렬 가능**해야 한다.
- **초당 10,000개**의 ID를 만들 수 있어야 한다.

## 2단계: 개략적 설계안

분산 시스템에서 유일성이 보장되는 ID를 만드는 방법은 여러 가지. 다음과 같은 선택지를 살펴본다.

- 다중 마스터 복제
- UUID
- 티켓 서버
- 트위터 스노플레이크 접근법

---

### 다중 마스터 복제

> k개의 데이터베이스가 있다면 각각의 DB의 id는 k만큼 증가한다.

| DB | 생성하는 ID 수열 |
|---|---|
| `db_1` | 1, 1+k, 1+2k, 1+3k, … |
| `db_2` | 2, 2+k, 2+2k, 2+3k, … |
| `db_3` | 3, 3+k, 3+2k, 3+3k, … |
| … | |
| `db_k` | k, 2k, 3k, 4k, 5k, … |

| 장점 | 단점 |
|---|---|
| 단순한 방법으로 ID 유일성 보장하면서 규모 확장성 어느 정도 해결 | 여러 데이터 센터에 걸쳐 규모를 늘리기 어렵다 |
| | ID를 시간순으로 정렬할 수 없다 |
| | 서버를 추가/삭제할 때 처리가 복잡 |

#### 참고 — MySQL 설정

> 특정 테이블에서만 증가 값을 변경하는 것은 불가능. 특정 테이블만 핸들링하고 싶다면 trigger를 통해서 핸들링.

```sql
-- AUTO_INCREMENT 값을 k로 설정
SET @@auto_increment_increment=k;
```

### UUID

> UUID는 유일성이 보장되는 ID를 만드는 또 하나의 간단한 방법. **128비트**이며, 충돌 가능성이 지극히 낮다.

> 위키피디아에 따르면 중복 UUID가 1개 생길 확률을 50%로 끌어올리려면 **초당 10억 개의 UUID를 100년 동안 연속해서** 만들어야 한다.
> https://en.wikipedia.org/wiki/Universally_unique_identifier

UUID 값은 `09c93e23-53j4-346d-bg9a-s24n3452njs2`와 같은 형태를 띤다.
<!-- TODO: 실제 UUID는 16진수(0-9, a-f)만 사용. "j", "g", "n", "s"는 hex가 아니므로 잘못된 예시. 책 원문 또는 OCR 오류로 추정. 올바른 예: 09c93e23-ce31-4a72-bda6-d11317c8ec64 같은 형태. -->

> UUID는 **서버 간 조율 없이 독립적으로 생성 가능.**

<!-- ![](images/p02-002.png) -->

| 장점 | 단점 |
|---|---|
| 서버 사이의 조율이 필요 없으므로 동기화 이슈가 없음 | ID가 128비트로 길다 (요구사항 미충족) |
| 각 서버가 자기가 쓸 ID를 알아서 만드는 구조 → 규모 확장이 쉬움 | ID를 시간순으로 정렬할 수 없음 |
| | ID에 숫자가 아닌 값이 포함될 수 있음 |

### 티켓 서버

> 핵심: `auto_increment` 기능을 갖춘 **데이터베이스 서버 1대를 중앙 집중형**으로 사용.

<!-- ![](images/p02-003.png) -->

| 장점 | 단점 |
|---|---|
| 유일성 보장 + 숫자로만 구성된 ID | 티켓 서버가 **SPOF**가 된다 |
| 시간순 정렬 가능 | 여러 대로 구축하면 데이터 동기화 이슈 발생 |
| 구현이 쉬움 → 중소 규모 애플리케이션에 적합 | |

### 트위터 스노플레이크 접근법

> 스노플레이크는 트위터에서 공개한 ID 생성 기법. 위에서 살펴본 방식들 중에는 제대로 요구사항을 충족하는 방식이 없었다. **그러나 이 방식은 충족할 수 있다.**

<!-- ![](images/p03-004.png) -->

**64비트 구성**

| 영역 | 비트 | 설명 |
|---|---|---|
| 사인(Sign) 비트 | 1 | 지금은 쓰임새가 없다. 나중을 위해 유보 |
| 타임스탬프 | 41 | 기원 시간 이후 몇 밀리초가 경과했는지 |
| 데이터센터 ID | 5 | → 32개의 데이터센터 지원 |
| 서버 ID | 5 | → 데이터 센터당 32개의 서버 지원 |
| 일련번호 | 12 | 각 서버에서 ID 생성마다 1씩 증가. 1ms 경과 시 0으로 초기화 |

## 3단계: 상세 설계

> 트위터의 스노플레이크 접근법을 사용해서 상세 설계.

<!-- ![](images/p03-005.png) -->

- **데이터센터 ID와 서버 ID**: 시스템 시작 시 결정. 일반적으로 운영 중에는 바뀌지 않는다.
  - 잘못 변경하면 **ID 충돌** 발생.
- **타임스탬프 / 일련번호**: ID 생성기가 돌고 있는 중에 만들어지는 값.

### 타임스탬프

> ID 구조에서 가장 중요한 **41비트**. 시간 순으로 정렬 가능하다.

<!-- ![](images/p03-006.png) -->

> 41비트로 표현 가능한 타임스탬프의 최댓값은 **약 69년**. 따라서 이 ID 생성기는 69년 동안만 정상 동작.
>
> 기원 시각을 현재에 가깝게 맞춰서 오버플로가 발생하는 시점을 늦춰 놓은 것. 69년이 지나면 기원 시각을 바꾸거나 ID 체계를 다른 것으로 이전해야 한다.

### 일련번호

- 일련번호는 12비트이므로 **4096개**의 값을 가질 수 있다.
- 어떤 서버가 **같은 밀리초** 동안 하나 이상의 ID를 만들어낸 경우에만 0보다 큰 값을 갖게 된다.

## 4단계: 마무리

- **시계 동기화(clock synchronization)**: 각각의 서버는 시간을 동기화해야 한다. 일반적으로 **NTP** 사용.
- **각 부분의 길이 최적화**: 필요에 따라 각 부분의 비트를 조절 가능. 69년보다 더 긴 수명이 필요하다면 타임스탬프 비트수를 늘릴 수 있다.
- **고가용성**: ID 생성기는 필수 불가결 컴포넌트이므로 높은 가용성을 제공해야 한다.
