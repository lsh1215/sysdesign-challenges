## 2. 컴포넌트 식별 (개략적 설계)

각 컴포넌트를 한 사람씩 답하는 식으로 식별.

| #   | 컴포넌트                   | 답한 사람 | 책 표현                         |
| --- | -------------------------- | --------- | ------------------------------- |
| 1   | 시작 URL 집합              | 보성      | Seed URLs                       |
| 2   | 미수집 URL 저장소          | 보성      | URL Frontier                    |
| 3   | HTML 다운로더              | 종민      | HTML Downloader                 |
| 4   | 도메인 변환기 (DNS)        | 종민      | DNS Resolver                    |
| 5   | 컨텐츠 파서                | 유경      | Content Parser                  |
| 6   | 중복 컨텐츠 검증           | 유경      | Content Seen? + Content Storage |
| 7   | URL 추출기                 | 민규      | URL Extractor                   |
| 8   | URL 필터                   | 민규      | URL Filter (부적절 URL 차단)    |
| 9   | 이미 방문한 URL 검사       | (논의)    | URL Seen?                       |
| 10  | 방문한 URL 저장소          | (논의)    | URL DB                          |
| →   | 새 URL을 다시 미수집 URL로 | 유경      | URL Frontier 재투입             |

### 2.1 중복 컨텐츠 검증 — S3 + NoSQL KV (드릴다운 미리)

윤기 발언 ("hash 충돌 비교 방법") 을 시작점으로, 작성자가 핵심 보강:

**Key/Value 구조**

- **Key** = 컨텐츠 본문의 SHA-256 해시값
- **Value** = S3 저장소의 URL (컨텐츠 실제 저장 위치)

**왜 이 구조인가** (민규 보강): 거대한 HTML 텍스트를 DB에 생짜로 저장하지 않음. S3 같은 별도 스토리지에 저장하고, KV 쪽엔 주소만 저장.

```
[컨텐츠 파서] → SHA-256 → [KV store]에서 hash 조회
                              │
                              ├─ HIT  → 폐기 (이미 있음)
                              └─ MISS → S3에 본문 업로드
                                        + KV에 (hash, S3 URL) 추가
```

### 2.2 URL 필터의 진짜 역할

처음 김경 / 참석자 의견: "이미 있는 URL 검사" 로 이해.

작성자 정정: URL 필터는 **부적절한 URL 차단** (안정성 고려: 악성 사이트, blocklist 등). "이미 방문" 검사는 다음 단계인 **URL Seen** 컴포넌트에서 처리.

## 3. 상세 설계 (Drill Down)

### 3.1 BFS vs DFS

민규 답변: **BFS 선택**.

| 알고리즘 | 평가                                              |
| -------- | ------------------------------------------------- |
| DFS      | 링크 깊이가 매우 길면 무한히 파고들 위험 → 부적합 |
| BFS      | 너비 우선이라 적합. 단 두 가지 문제 존재          |

**BFS 의 두 문제** (민규):

1. **예의 위반 (politeness)**: 같은 도메인 URL이 한 페이지에서 많이 추출 → 그 도메인 폭발적으로 때림
2. **우선순위 부재**: 모든 URL 동등 취급 (중요도/트래픽/빈도 차등 없음)

→ 두 문제 모두 **URL Frontier 내부 구조** 로 해결 (§3.2).

### 3.2 URL Frontier 내부 구조

```
URL 입력
    ↓
[순위 결정장치 (Prioritizer)]   ← URL의 우선순위 계산
    ↓
큐 high / mid / low (우선순위별 분리, 각각 FIFO)
    ↓
[큐 선택기 (Front Queue Selector)]   ← weighted random pick
    ↓
[큐 라우터 (Back Queue Router)]      ← 도메인별 매핑 테이블
    ↓
큐 1 / 큐 2 / 큐 N (도메인별, 각각 FIFO)
    ↓
[큐 선택기]
    ↓
작업 스레드 (도메인별 1개)
```

**상단 (front queue)**: priority 차등 처리 (어떤 URL을 먼저 처리할지).
**하단 (back queue)**: 도메인별 분리 (politeness — 같은 도메인은 한 스레드만).

#### 우선순위 결정 방식 (종민)

> 미세 튜닝 영역. 예: "Apple" 키워드 → 공식 apple.com 사이트가 단순 언급 사이트보다 더 가치 있음. **인위적으로 우선순위 보정**.

작성자가 GPT 검색해본 결과 관련 논문 2개 있음. 수식이 복잡해서 점프. "별도 엔지니어 영역" 으로 두고 넘어감.

#### 큐 종류 의문 (작성자)

> "우선순위 결정 후 들어가는 큐는 PriorityQueue? 아니면 그냥 FIFO?"

**결론: FIFO 큐**.

- 큐 자체가 priority 별로 이미 분리됨 (high 큐 / mid 큐 / low 큐)
- 각 큐 내부는 FIFO
- 큐 선택기가 가중치 기반 random 으로 뽑아감 (예: high 60% / mid 30% / low 10%)
- 우선순위 결정장치는 **어떤 큐에 넣을지** 만 결정

### 3.3 신선도 체크 — 왜 따로 필요한가

#### 작성자 의문

> "어차피 SHA로 컨텐츠 dedup 하잖아. URL 같아도 컨텐츠 바뀌면 hash 달라져서 새로 저장될 텐데, 신선도 체크 왜 따로?"

#### 종민 답변 (정답)

URL Seen 검사 (블룸 필터) 가 먼저 막아버림. Bloom filter HIT → "이미 봤음" → 폐기 → URL Frontier 큐에 안 들어감 → **다운로드 자체가 일어나지 않음** → 컨텐츠 dedup 검사 기회조차 없음.

> "URL만 동일하면 단순히 컨텐츠가 바뀌어도 인지할 방법이 없을 것 같아요." — 종민

→ 작성자: **"정답. 정답. 이거죠."**

#### 신선도 batch 의 역할

별도 batch job으로:

- "X일 지난 URL 은 visited 라도 큐에 강제 enqueue"
- 다운로드 → SHA-256 비교
  - hash 같음 → 본문 그대로 → 저장 안 함, last_crawled 갱신
  - hash 다름 → 본문 변경 → 새 버전 저장

#### 책의 최적화 전략 (작성자 언급)

- 페이지 변경 이력 기반 (정적/동적 차등)
- 우선순위 기반 (high-priority 자주)
- HTTP `Last-Modified` / `ETag` (304 Not Modified 시 다운로드 스킵)

### 3.4 블룸 필터의 영속성 — RedisBloom

#### 작성자 의문

> "Redis 메모리 기반인데 배치 작업 끝나면 메모리 날아가지 않나? Bloom filter 휘발 안 되나?"

#### 결론

**휘발 안 됨**. Redis 자체 영속성:

- **AOF (Append-Only File)**: 매 write 명령을 로그로 append
- **RDB snapshot**: 일정 주기마다 메모리 → 디스크 dump

배치 작업 다음 회차 시작 시 Redis 가 디스크에서 복구 → Bloom filter 그대로 살아남음. 그래서 신선도 체크가 필요 (위 §3.3).

### 3.5 DNS 캐싱 (DNS Resolver) — 작성자 + 종민 공통 의문

> "OS 레벨에서 기본적으로 DNS 캐싱하는데 (hosts 파일 등), 굳이 별도 리졸버 컴포넌트 만들 정도의 요구사항은 뭘까?"

명확한 결론 못 냄. **참고: 보강 필요 항목.**

추정: 로컬 캐시만으로 부족할 정도로 도메인 다양성이 큰 경우, 별도 분산 DNS resolver 필요.

책이 강조한 부분 (작성자):

- **지역성 (locality)**: 리전별 운영 → 안정성 + 지연 단축
- **타임아웃**: DNS 조회 타임아웃 → 안정성

### 3.6 확장 모듈 — 새 컨텐츠 형태 지원

요구사항에 명시되진 않았지만 책에서 언급:

```
URL 추출기 (인터페이스)
    ├── HTML 추출기 (현재 구현)
    ├── PNG / 이미지 추출기  ← 추가 시 인터페이스 구현체만 추가
    ├── PDF 추출기
    └── 모니터 (저작권 / 상표권 침해 모니터링)
```

## 4. 다루지 못한 / 추가 검토 항목

### 4.1 동적 렌더링 (서버 사이드 vs 클라이언트 사이드)

종민 제기 (책 161p):

> "URL에서 바로 HTML이 안 나올 수 있다. 요즘 React 기반 클라이언트 사이드 렌더링 사이트 많음. 미수집 URL Frontier ↔ HTML 다운로더 사이에 _렌더링 컴포넌트_ 가 한 개 더 필요하지 않을까?"

작성자 동의. **추가 컴포넌트 후보로 남김:**

```
[미수집 URL Frontier]
        ↓
[렌더러 (headless browser, 예: Puppeteer)]   ← 추가 검토
        ↓
[HTML 다운로더]
```

### 4.2 모니터링 (다음 회차에 반영 예정)

웹 크롤러에서 모니터링 대상 (작성자 + 참석자):

- **DB 모니터링** — 특히 블룸 필터 (Redis 메모리 / saturation)
- **컨텐츠 파싱 에러** — 파싱 에러 많이 날 듯 → 로그로 잡기
- **큐 적체** — 스레드 처리 속도 < 큐 enqueue 속도 시 이상 신호
- **중복 처리 race** — 같은 URL 동시에 여러 번 처리되는 경우

또 책의 확장 모듈에 "웹 모니터" 가 있는데, 이건 저작권/상표권 침해 모니터링 으로 추정.
