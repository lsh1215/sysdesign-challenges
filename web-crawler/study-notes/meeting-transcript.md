## 1. 문제 정의 (Clarifying)

**작성자**: 먼저, 문제 정리 파트에서는 저희가 해야 되는 문제 정리가 다 끝나고 가야 된다고 보거든요. 근데 아직 문제 정의가 끝나지 않아서 문제 정의를 좀 더 하고 갈게요.

원래 이게 functional하고 non-functional 나눠서 하긴 하는데, 아직 그런 준비는 안 돼 있으니까 그냥 저희가 하던 것처럼 마구잡이로 요구사항 정리하고, 거기서 정량화된 거랑 비정량화된 걸로 나눠보는 식으로 할게요.

처음에 저희가 정리한 게 — 검색엔진 인덱싱용이고, 웹페이지 수는 이 정도 되고, 신규/수정된 페이지도 포함해야 되는지, 보존 기간은 5년이어야 되고, 중복 컨텐츠 제거되어야 되는지, 이런 것까지는 했고요.

그리고 확장성. 규모 확장성. 병행성 활용해서 좀 빨리 처리해야 된다 이런 얘기였던 것 같고. 안정성은 약간 악성 페이지가 들어가면 안 된다 이런 거였고. 그리고 추가됐던 요구사항들이… 저도 읽어야 해서.

예절(politeness). 수많은 리퀘스트 관련해서, 약간 DoS 공격 의심받을 수 있으니까 이런 거 피해야 되고. 그리고 확장성 — 새로운 형태의 컨텐츠도 고려된 시스템을 만들어야 합니다.

일단 이게 책에서 가지고 있던 요구사항 정리 관련해서, 저희가 인터뷰어한테 뽑아내야 되는 정보였고, 그러니까 질문을 해서 이런 거 뽑아냈다 정도까지 했고요.

그 다음에, 이걸 바탕으로 어떻게 좀 더 추가를 해야 되냐 했을 때는… 그러면 QPS 같은 거. 월에 1B이 맞나요? 잠시만요.

**참석자**: 10억.

**작성자**: 매월 1 billion이면 이게 초당 몇 건의 페이지를 처리해야 되냐. 좀 나눠 놓을게요. 잠시만요. 저희가 원래 하던 방식이 아니라 약간 조금 다른 방식으로요.

그래서 인터뷰어한테 뽑아내는 정보가 위에 거고, 아래는 우리가 뽑아내야 하는 정보예요. QPS가 이 정도 돼야 된다 — 초당 몇 건의 페이지를 처리해야 되냐.

이거에 대해 추론을 어떻게 하나요? 종민님?

**종민**: 저는 10억 건이니까, 초당으로 나누려면… 이게 월이죠. 월이니까 30일로 나눈다고 가정을 하고, 그 다음에 24로 나누고, 1시간이 3600초니까 또 나누고. 이렇게 다 나눠보면 책에서는 대략 400 pages/sec로 가정했습니다.

**작성자**: 네, 맞아요. 이런 식으로 했죠. 1B을 86,400초/일 × 30일 = 2,592,000 으로 나누면 **약 400 pages/sec** 나오는 거고. 책에서도 그렇게 잡았고요. 물론 대략적인 거죠.

그러면 다음, 평균 페이지 크기는 책에서 몇으로 잡았었죠?

**참석자**: 500KB입니다.

**작성자**: 평균 500KB였죠. 그러면 월에 어느 정도 데이터 용량이 쌓인다고 봐야 되나요? 월에 10억 페이지를 모으니까…

**참석자**: 500KB × 10억 하면, **500TB**가 필요합니다.

**작성자**: 이것도 약식이죠. 원래는 2의 제곱수로 1024 × 1024 이렇게 가는 거잖아요. KB → MB → GB → TB. 근데 뒤에 천에서 천에서 천 이렇게 해서 약 500TB가 나온다. 맞죠? 여기까지 다 따라오시고 계시죠?

**참석자**: 네.

**작성자**: 그러면 데이터 보존 기간이 5년이었죠. 5년으로 잡으면 얼마 정도 데이터 용량이 필요한가요?

**참석자**: 500TB에 12개월 곱하고, 거기다 5년이니까 5 곱한다 치면, 500TB × 60 이니까…

**작성자**: 처음에는 37PB가 나왔는데, 단위가 천 단위로 올라가니까. **30PB**로 정정하겠습니다. 맞죠?

**참석자**: 네 맞습니다.

**작성자**: 지금 이 정도 했던 것 같고요. 혹시 더 있었나요? 책에서 잡았던 거 중에.

**참석자**: 책에서 peak QPS도 정의를 해놨어요.

**작성자**: 아! peak QPS 정했었죠. peak QPS는 그냥 평균에 2 곱한 거였죠? 책에서 대략 두 배로 잡더라고요. 그래서 **약 800 QPS**. 맞죠?

**참석자**: 네, 맞아요.

**작성자**: 이게 저희가 비기능 요구사항을 토대로 계산 가능해서 잡아야 하는 정보. 이 정도가 나왔던 것 같아요. 맞죠?

**참석자**: 맞습니다.

**작성자**: 문제 정의 단계에서 이 정도 정의를 하고 인터뷰어한테 정보 뽑아낸 다음에 넘어가야 하는 거라고 저는 봤어요.

## 2. 컴포넌트 식별 (개략적 설계)

**작성자**: 그러면 이제 이 데이터 정보를 가지고 어떤 컴포넌트가 필요할지 추정을 좀 해봐야 될 것 같거든요. 그래서 컴포넌트가 뭐가 필요한가를 논의해봐야 돼요.

처음에 필요한 컴포넌트가 뭐라고 보시나요? 보성님?

**보성**: 우선 탐색해야 되는 대상이 되는 URL.

**작성자**: 네, 탐색할 대상 URL. 약간 좀 더 디벨롭해서 설명을 하면 시작 URL로 봤던 것 같아요. 제가 처음에 추측하기로는. 그렇죠?

**보성**: 네, 맞아요.

**작성자**: 그러면 그 다음 컴포넌트는 뭐가 필요한가요?

**보성**: 미수집 URL 저장소가 필요합니다.

**작성자**: 그렇죠. 미수집 URL 저장소. 이게 아마 URL Frontier 였나요?

**보성**: 네, 맞아요.

**작성자**: 그러면 그 다음 컴포넌트는 뭐가 필요할까요? 종민님, 두 개 정도 말씀해주시면 될 것 같은데.

**종민**: 그 다음은 URL에서 HTML을 다운로드 받는 다운로더가 필요합니다. 그리고 한 개 더하면, 책에 있는 도메인 리졸버 말씀이신가요?

**작성자**: 네, 맞아요. 여기서 다음 단계로 넘어가야 되는 게, HTML 다운로드를 할 때 그냥 되는 게 아니죠. 어떤 URL을 가지고 할 때, 도메인 변환기가 필요해요. 도메인 이름 가지고 IP 찾기. 맞죠?

**종민**: 맞습니다.

**작성자**: 그러면 그 다음 컴포넌트는 뭐가 필요할까요? 유경님.

**유경**: 다운로드한 다음에는 파싱하고 검증할 수 있는 콘텐츠 파서가 필요합니다.

**작성자**: 콘텐츠 파서. 그러면 그 다음에 하나 더 불러주세요.

**유경**: 그 다음에 중복된 컨텐츠가 있는지 검증하는 중복 컨텐츠 검증 컴포넌트가 필요합니다.

**작성자**: 그리고 중복 컨텐츠 검증을 어디서 어떻게 가져올까요? 이거는 사실 드릴다운 단계에서 다루는 건데 좀 더 디벨롭해서 여기서 만지고 갈게요. 중복 컨텐츠 검증을 뭘로 어떻게 해야 할까요?

**참석자**: 해시…

**작성자**: 해시값을 비교해서. 책에서 그렇게 나와 있죠. 이거에 대해서 설명이 가능해야지 중복 컨텐츠 검증이 왜 필요한지, 어떻게 할지가 나와요. 사실 그래야 이 컴포넌트가 필요한지가 정해진다고 생각하거든요. 이걸 조금 설명 가능하신 분 있을까요?

콘텐츠 저장소에서 중복 콘텐츠가 있는지만 확인해서 다음 단계로 넘어가야 되는데, 중복 콘텐츠 검증을 어떻게 한다는 걸까요? 윤기님? 뭔가 아실 것 같은데.

**윤기**: 되게 어려운 걸 드네.

**작성자**: 아, 어려운 거긴 하지.

**윤기**: 중복 콘텐츠 검증을 하기 위해 기존에 있는 본문의 해시 값을 비교해서 찾아낸다고 하긴 하는데… 형이 지금 알 수 있는 정보는 hash 충돌 비교 방법, 약간 그런 거지. hash 확인하고, hash code 같으면 `equals` 비교하는 그런 식.

**작성자**: 그럼 여기서 하나만 들어갈게요. hash key 값이 뭐고 value가 뭔지. hash store면 NoSQL이라는 소리고, 그러면 NoSQL이면 key/value가 있잖아요. key/value가 뭔지 한번 설명 가능하실 분이 있으실까요? 이것 때문에 컴포넌트가 하나 더 필요할 수도 있어서요.

사실은 나중에 좀 드릴다운에서 나와야 되는 건데 제가 조금 설명드리면, **키가 해시값**이고 — **value는 사실 S3 저장소 URL**이 될 거고. 이게 왜 이렇게 되냐면, 컨텐츠를 저장할 때 HTML 컨텐츠 파싱해서 나온 거대한 텍스트 형태의 덩어리를 저장해야 되잖아요.

근데 이걸 생짜로 저장할 수가 있나요? DB에 생짜로 저장을 하나요?

**참석자**: 어... 정적 파일을, 정적 HTML 파일 같은 걸 파일을 생으로 DB에 저장을 한다? 할 수도 있죠.

**작성자**: 할 수도 있는데, 일반적으로 어떻게 하죠? 민규님 어떻게 할까요?

**민규**: 오로지 저 애들만 저장하기 위한 다른 스토리지가 있는 게 좋겠죠. 그리고 KV 쪽에는 주소만 갖다 놓고.

**작성자**: 네. 그래서 정적 컨텐츠 같은 건 S3 같은 데에 저장을 하고, S3 저장소 URL이 value가 되고, 이 컨텐츠를 가지고 해시 돌려서 나온 해시값이 key로 쓰이는, 그런 구조가 되는 거예요. 그래서 DB가 하나가 아니라 S3 같은 클라우드 스토리지 컴포넌트가 하나 더 들어와야 되는 거죠.

여기까지 다 이해되신 거죠?

**참석자**: 네.

**작성자**: 그럼 저희가 중복 컨텐츠 판단까지는 마친 상태예요. 해시 값으로 컨텐츠 잘 저장하고, 해시 값으로 중복 컨텐츠 검증까지 마쳤어요. 그럼 그 다음 단계가 뭐가 필요한가요? 민규님 한 번 더.

**민규**: HTML 안에 있는 링크들을 추출하는 컴포넌트가 필요합니다.

**작성자**: 좋습니다. 내부 URL 추출기. 그러면 추출 마치고 나서 다음 작업이 뭔지 알려주세요. 지금 내부 URL 추출까지 맞췄어요. 중복 컨텐츠 검증해서 — 중복이면 저장 안 하고, 새 거면 저장한 다음에 내부 URL 추출했죠. 다음에 해야 될 거.

**민규**: 이제 골라낸 URL을 URL 필터로 보냅니다.

**작성자**: 이거 왜 하는 거죠?

**참석자**: 중복할… 이미 있는 건지 알지.

**작성자**: 과연 그럴까요? 이거 왜 필터하는 거죠?

**참석자**: 그… 필터링 같은 거 관리하는 걸로 알고 있어요.

**작성자**: 아 좋습니다. 부적절한 URL 제외. 들어가면 안 되는 거. 아까 저희가 안정성 고려를 했었잖아요. 거기서 필터를 하는 거고. 김경님이 말씀하신 "이미 방문한 거 검사" 는 사실 이후 처리 단계에서 하는 거기 때문에 — 그것도 필터라고 보긴 하지만, 컴포넌트 단위로 했을 때는 "이미 방문한 URL 저장소" 가 더 맞아요.

방문한 URL인지 판단하려면 또 뭐가 필요하죠?

**참석자**: 중복… DB가 필요하죠. DB에서 뭔가 중복 갖고 있는 URL인지 아닌지 판단을 하겠죠.

**작성자**: 이건 이따가 더 만지면 되고. 그럼 만약에 중복이 아니에요, 방문한 URL이 아니에요. 그럼 어떻게 해야 되나요? 유경님, 마무리해주시죠.

**유경**: 미수집 URL로 보냅니다.

**작성자**: 좋습니다. 이게 아니면 미수집 URL로 다시 보내서. 이게 책에 나왔던 컴포넌트 그림인 것 같아요. 맞죠? 어떤 내부적인 과정을 거치는지까지도 어느 정도 알아봤고.

## 3. 상세 설계 (Drill Down)

### 3.1 BFS vs DFS

**작성자**: 그러면 이제 상세 설계로 들어가야 되는 거죠. 처음에 나왔던 게 BFS, DFS 얘기였던 것 같은데 맞나요?

**참석자**: 맞습니다.

**작성자**: 컴포넌트 단위로 크게 알아봐야 되는데, 저희가 가진 URL들을 탐색을 할 때 — 시작 URL은 그냥 검사하면 되는 거고, 프론티어 단계에서 갖고 있는 URL들을 검사해 나갈 때 DFS로 할지 BFS로 할지가 중요하잖아요. 민규님이 아까 잘 알고 계셨던 것 같아서. BFS로 해야 하는지 DFS로 해야 하는지, 그 이유 설명 가능하실까요?

**민규**: 일단 URL을 DFS로 하게 되면, 만약 링크 깊이가 매우 길어지면 정말 무한히 파고들 위험이 있어서 DFS는 적합하지 않고, **너비 우선 탐색인 BFS가 좀 더 웹 크롤러에 적합**하다고 생각해요. 아마 책에서 BFS의 두 가지 문제점 이야기했었던 거 같은데. 저희가 아까 이야기했던 예의 위반(politeness)이 있을 수 있고, BFS가 결국 모든 URL을 동일하게 취급하다 보니까 우선순위에 대한 게 없어서, 탐색 대상 URL의 중요도에 대한 지정이 필요하다 — 그거에 대해서 책에서 이야기했던 거 같아요.

**작성자**: BFS는 선택을 한 이유가 URL 깊이가 너무 깊어졌을 때 못 다루는 거고. 그래서 BFS가 선택이 됐는데, **두 가지 문제점이 — 하나가 예의(politeness)**. URL을 막 때리는 거. **두 번째 문제가 우선순위가 없다**, 였죠. 중요도를 매겨서 트래픽이라든지 빈도 등을 체크해서 우선순위를 따를 필요가 있다.

예의 관련된 거는 아마 DoS 였던 것 같아요. 특정 URL에 너무 많이 때려박으면 그쪽 입장에서는 너무 많은 요청이 하나의 IP에서 오는 거고, 그쪽에서 rate limit 걸어놨으면 안 될 거고. 보통은 robots.txt 가이드라인을 지킬 필요가 있다. 그 다음에 나왔던 내용이… 잠시만요, 저도 책 보고 따라갈게요.

웹사이트 시간 차를 두고. Crawler 얘기가 나오네요 그 다음에.

### 3.2 URL Frontier 내부 구조 (큐 라우터, 큐 선택기, 우선순위)

**작성자**: 상세 컴포넌트를 좀 더 그려봐야 될 것 같은데. 저희가 스레드를 그래서 나누잖아요. 그 스레드가 큐에 들어가야 되는 애들이 어쨌든 작업을 하나씩 할 때, URL도 다 지정을 해서 특정 큐에만 넣는 걸로 알고 있어요. 제가 이해한 게 맞나요? 책에서.

큰 컴포넌트가 있고 아래에 작은 세부 컴포넌트가 있어요. 큐 라우터가 있었던 것 같고, 라우팅이 되니까 큐들이 좀 있는 걸로 기억해요. 큐 선택기 통해서 작업 스레드 결정이 되는.

이게 어떤 URL에 대해서는 어떤 스레드가 작업하게, 큐에서 별도로 딱 지정돼서 작업할 수 있게 하는 거죠. 그래서 스레드가 별도의 FIFO 큐를 갖고 있는 식의 작업 방식이었던 것 같아요.

#### 우선순위 결정

**작성자**: 위에서 이제 우선순위를 먼저 결정하고 그 이후에 큐 라우팅을 한다 — 책에서 그렇게 말했던 거 같아요. 우선순위 결정을 해야 되는데, 키워드가 같다고 그런 홈페이지에 대한 중요도가 똑같진 않다. 우선순위를 결정해서 선택해야 한다. 였는데, 저는 사실 뭘 우선순위로 한다는 건지 잘 이해를 못 해서요. 이거 혹시 이해되신 분 있을까요?

**종민**: 그게 미세한 튜닝이 되어야 하는 부분이라 저는 개인적으로 생각하는데, 책에서 봤을 때 예를 들어 "Apple" 이라는 키워드가 있다 치면, 동등한 우선순위를 두면 단순히 공식 Apple 사이트를 긁어오는 게 더 가치 있는 자료들인데, 단순히 Apple을 언급한 자료를 긁어오는 거에 대해서 우려를 해서, **인위적으로 좀 더 우선순위를 보정**한다고 이해를 하고 있습니다.

**작성자**: 그거에 대해서 제가 잠깐 GPT에 자료 찾아봤었는데 논문이 두 개 정도 있더라고요. 근데 이게 수식이 좀 복잡했습니다. 그러면 그거는 점프하고 가겠습니다. 그런 수식까지 다 알 필요는 없으니까.

어쨌든 우선순위 결정에 대해서는 별도의 엔지니어가 안 된다 — 이렇게 하고 넘어가야 할 것 같네요. 논문상 나와 있는 거 보면 거의 거기서 나온 우선순위 결정 방법을 따라갈 거고, 그렇게 결정해서 알맞은 큐에 넣는다 — 이거인 것 같은데.

#### 의문: PriorityQueue인가 FIFO Q인가

**작성자**: 그러면 여기서 저는 조금 의문이었던 게, 우선순위 결정을 먼저 하고 라우팅을 하잖아요. 그러면 우선순위 결정 후 들어가는 큐는 FIFO Q인가요, 아니면 PriorityQueue인가요? 우선순위 큐인지 그냥 생 큐인지. 이게 좀 궁금했거든요.

**참석자**: 큐 선택기가 어느 정도 빈도수나 이런 걸 가중치로 적절히 계산해서 가져오는 걸로 이해를 했습니다.

**작성자**: 맞아요. 같은 priority 안에서는 체크를 안 하는 방식이에요. 큐가 다른 거예요 큐가. 큐가 우선순위가 high, mid, low 이렇게 나오는 거고, **그 내부 큐는… 그러니까 한번 걸쳐서 나오는 애니까 그냥 FIFO 큐**인 거죠.

이해되죠? 종민님도 그렇게 이해하시고 저한테 말씀해주신 것 같은데.

**종민**: 맞습니다.

**작성자**: 좋습니다. 그래서 여기서 결정해서 — 큐 선택기 정도가 위아래로 다 있어야 되니까 위에서 하나 추가되는 정도. 사실 큐 선택기라는 게 어쨌든 큐 여러 개 있는 것 중에 작업 할당하려면 선택해서 넣어야 되는 거니까, 매번 다 있어야 되는 아주 유의미한 컴포넌트는 아니라고 보고. 이렇게 거의 완성이 됐을 거예요.

여기서 매핑 테이블이 빠진 게 있는데, 그거는 당연히 큐 라우팅하려면 있어야 되는 거여서 별도로 안 넣었고요. 어쨌든 우선순위하고 예의(politeness) 어떻게 할 건지 그것까지 알아본 것 같아요. 맞을까요?

**참석자**: 근데 어떤 게 우선순위 큐고 어떤 게 그냥 큐라는 말씀이시죠?

**작성자**: 둘 다 FIFO 큐예요. 제가 말했던 건, 우선순위 결정을 하는 건데, 우선순위 결정을 해서 나온 이 큐들도 PriorityQueue였던 건지 궁금했던 거예요. 우선순위 결정을 해서 그냥 큐에 넣고 거기서 PriorityQueue로 뭔가 왔다 갔다 이동하는 건지, 아니면 이 큐 자체가 좀 다른 큐인 건지. 그게 궁금했던 건데, 결론적으로 — 이 큐 자체가 이미 분류가 돼 있는 큐다, 이 말이야. 이게 high 큐고, 이게 low 큐, 이런 거지. 이해됐지?

아래에서 우선순위 결정을 하는 거는 어떤 큐에 넣을지를 고민하는 거고. 그래서 위아래 거는 그런 게 되는 거고, 아래에서는 URL을 가지고 어떤 URL은 어디다 넣겠다 — 이렇게 되는 거지. 이해됐지?

**참석자**: 네, 오케이.

**작성자**: 그러면 큐 라우팅하고 큐 우선순위 선택하는 거 — 이거 다 이해 끝난 거 같아요. 여기서 혹시 더 이해 안 되는 거 있는 사람 있을까요? 유경이처럼 물어봐 줘도 돼요. 다들 괜찮으신 거 같네요.

### 3.3 신선도 체크 + 블룸 필터

**작성자**: 그래서 마지막에 했던 게 신선도 체크. 그거였던 것 같아요. 제가 신선도 체크 왜 하는지가 궁금했는데, 왜 궁금했냐면 — 부가적인 얘기로 — "이미 방문한 URL인지 체크" 어떻게 하는지 혹시 아시나요? 책에서.

오른쪽 위 컴포넌트를 봐주셔야 돼요. 민규님? 아 화면 공유 전에 했던 거. 네 거기 따라주시면 됩니다.

자 그럼 여기로 넘어갈게요. 왜냐하면 아까 S3 저장소가 필요하다 이런 얘기 했던 것처럼, 드릴다운 한다는 건 좀 더 세부적으로 알고 넘어가는 거니까. 저희가 얘기 안 한 게 있어요. 중복 컨텐츠 검증하고 이미 방문한 URL 검증하는 부분에서 알고리즘 같은 걸 설명했었거든요.

위에서 설명했던 게 해시 방법이었던 것 같고, 이미 방문한 URL인지 여부에 대해 체크하는 게 뭐였을까요? 어떤 방법이었을까요? 기억나시는 분 있나요?

**참석자**: 책섬이라 했던 것 같은데. 해시도 있긴 하고.

**참석자**: 블룸 필터.

**작성자**: 네 **블룸 필터**였죠. 그래서 이 블룸 필터를 그러면 어떻게 하겠다는 건지 궁금해서. 왜냐하면 블룸 필터라는 게 한번 해시 돌리고 딱딱딱 뜨면 그거 집어넣고, 나중에 또 집어넣을 때 충돌이 다 되면 "아, 이거는 충돌 가능성이 너무 높다" 뭐 이런 식으로 판단하는 거잖아요.

블룸 필터로 했을 때 어떤 DB를 쓰고 어떻게 하는지 좀 궁금해서 제가 찾아봤었습니다. 찾아봤는데 **RedisBloom** 이라는 게 있대요. 찾아보면 RedisBloom 이라고 나오거든요. Redis에서 블룸 필터 써가지고 검사할 수 있게 하는 모듈인 거죠.

#### 신선도 체크 왜 하는가

**작성자**: 제가 궁금했던 게, 웹 크롤러를 배치 작업으로 돌린다고 했을 때 — 시작 URL은 우리가 설정해놨고, 미수집 URL을 한 번 거쳐서 모든 시작 URL이 들어가게 되고, 컨텐츠 파싱하고 중복 컨텐츠 검증하는 단계에서 — 중복 컨텐츠 검증을 어떻게 할지, 어떤 데이터를 갖고, 그 컨텐츠 데이터로 SHA 돌린 다음에 충돌하는지 보겠다 — 이런 거였잖아요.

근데 제가 궁금했던 건 신선도 체크 왜 하냐 이거죠. 그러니까 중복 컨텐츠 검증되면 된 거 아닌가? 어차피 컨텐츠가 바뀌었으면 상관없는 거 아닌가 하는 생각이 있었어요. 제 질문 이해되셨나요?

**참석자**: 네 이해됐습니다.

**작성자**: 다른 분들도 이해되셨나요? 너무 오래된 점, 다른 일을 하고… 그게 중복 컨텐츠 검증이 맞는데 신선도 체크 얘기인 건 맞는데. 신선도 체크 검사 자체를 왜 하냐는 거지. 중복 컨텐츠에서, 만약 내가 봤는데 컨텐츠 내용이 달라졌으면 해시값이 달라질 거고, 그러면 당연히 저장이 될 텐데 — 중복 처리 안 될 텐데, 중복이 아닌 걸로 처리될 텐데 — 왜 굳이 신선도 체크를 해야 되냐 이거지. 이거 왜일까요?

**참석자**: 제가 생각하기로는… 그 URL이 동일한데 데이터가 바뀌면 결국 새로 수집을 해야 하는데, 이 로직 보니까 이미 방문한 URL이면 버리는 걸로 되어 있거든요. 그래서 신선도 체크가 필요할 것 같습니다. **URL만 동일하면 단순히 컨텐츠가 바뀌어도 인지할 방법이 없을 것 같아요.**

**작성자**: **정답. 정답. 이거죠.** 이것 때문에 하는 거죠.

그러니까 우리가 중복 컨텐츠 검사를 하는 건 맞는데, 우리가 내부 URL을 추출한 다음에 이미 방문한 URL인지 검사를 한 번 돌리잖아요. 그때 이미 그 URL이 블룸 필터에 있으면 — "얘는 이미 수집된 건데?" 하고 그 이후에 검사를 안 돌리는 거죠. 그러면 중복 컨텐츠 검사 자체가 안 이루어지겠죠. 이 미수집 URL 프론티어 자체에 안 들어갈 테니까.

여기까지 다 이해 되신 거죠?

#### 블룸 필터 영속성

**작성자**: 제가 궁금했던 건 — Redis면 배치 작업 끝나면 메모리 있는 거 날릴 것 같은데, 이런 생각을 했거든요. 블룸 필터 안 걸릴 것 같은 생각을 했는데 그렇지 않더라고요. 안 날리더라고요.

**Redis에서 AOF나 RDB, 어쨌든 persistence 제공**하는 게 있잖아요. 그걸로 블룸 필터에 있던 걸 한번 저장해놓고 다음 배치 작업 시작될 때 다시 올리는 거죠. **결국 휘발되지 않는 거죠. 계속 남아 있다.**

휘발 안 되고 Redis에도 남아 있기 때문에, 이미 방문한 URL에 대해서는 — 종민님이 말씀하셨던 것처럼 — 검사 안 돌리니까 신선도 체크가 필요하고. 신선도 체크는 책에 나와 있던 것처럼 변경 이력을 확인한다든지, 아니면 중요한 거는 한 번 다시 돌린다든지, 그렇게 해서 큐에 enqueue 해주는 거죠. 우선순위 결정 큐에 다시 인큐해주는 거죠.

여기까지 다 이해 되신 거죠? 좋습니다.

### 3.4 분산 크롤링 + DNS 캐싱 + 확장 모듈

**작성자**: 사실 여기서 나왔던 내용에서 기본적인 요구사항에 있던 컴포넌트 설계나 디테일은 대부분 따라온 것 같아요. 그럼 이제 조금 남은 게 — 제가 기억하기로 — 분산 크롤링이라고 해서 성능을 높이겠다, 여러 서버에 부하를 분산하는 방법, 그거 하나 나왔던 거 같고. 근데 그건 컴포넌트로 그릴 만큼의 의미가 있지 않다고 생각해서 넘어갔고요.

DNS 캐싱. 변환 결과 캐싱. 이런 거 나왔던 것 같아요. 근데 이게 저는 약간 너무 추상적으로 적혀 있다고 해야 하나? 너무 내용 요약돼서 적혀 있어가지고. 혹시 이거 설명해주실 분 있나요?

**종민**: 저도 사실 궁금하긴 합니다. 왜냐하면 OS에서 기본적으로 DNS를 캐시하는데, 굳이 별도로 리졸버라는 컴포넌트를 만들 정도면 어느 정도 요구사항이 있어야 되는지, 그런 거에서 고민을 했는데 아직 저도 의문이 해소되지는 않았어요.

**작성자**: 오케이. 다들 비슷하게 생각하시는 거 같네요. 저도 종민님하고 똑같은 생각이었거든요. DNS라는 거 자체를 원래 캐싱하고 사용하잖아요, 저희가. 호스트 파일 같은 거 확인하고 그런 거잖아요.

google.com을 검색했을 때 무슨 일이 벌어지는가 했을 때, 무조건 IP 차드로 DNS에 박는 게 아니고, DB에 조회하는 게 아니고, 저희가 기본에 가지고 있던 캐싱 — 로컬에 캐싱해놓고, 호스트 파일에 적어놓고 — 그런 걸 먼저 다 확인 끝내고, 거기에 없다 하면 DNS에서 IP를 따오는 거거든요. 근데 이게 별도로 그런 과정을 거쳤음에도 DNS 리졸버가 필요한지, 좀 궁금하긴 해요.

근데 아마 그런 건 있을 것 같아요. 캐싱 정보가 그렇게 많지 않다면 뭔가 좀 필요할 수도 있겠다, 이런 생각은 있어요. 그래서 그건 그럴 수 있고.

지역성으로 로컬리하게 리전해서 운영하는 거하고, 타임아웃 정해놓는 거. 이거는 안정성 때문에 더 필요하다고 설명했던 것 같고. 나머지는 사실 더 괜찮은 것 같고.

여기서 하나만 더 — 확장성 쪽에서 하나 있었던 것 같거든요. 요구사항에는 안 적혀 있었는데, 데이터가 변경될 수 있다, 이미지 같은 데이터를 다운로드 받아야 할 수 있다, 라는 게 하나 있어서. 그래서 중복 컨텐츠 검사하는 그쪽에서 PNG 이미지 다운로더, URL 추출기, 모니터 — 이런 컴포넌트들을 확장 모듈로 분할해놨던 것 같아요.

필터 위에 있는 — 지금 갖고 있는 건 내부 URL 추출기 밖에 없는데, 여기에 이미지 추출기 같은 걸 놓는 거죠. 사실 아마 인터페이스로 구현해서 추출기라는 게 있고, 그 인터페이스의 구현체가 여러 개 있는 그런 느낌이 아닐까 라고 생각합니다.
